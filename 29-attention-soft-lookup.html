Attention I: The Soft Lookup

Every transformer you have ever run — every LLM, every cross-attention block in your ComfyUI graph — is built from the operation on this page. We are going to derive it, not memorize it: attention is a dictionary lookup with the sharp edges filed off until gradients can flow through it. Four pages cover the mechanism (soft lookup here, the $\sqrt{d_{\text{head}}}$ scale and heads on page 30, self vs cross on page 31, causal masking on page 32). This one earns the formula.

🧵 Thread [THREAD:dot-product beat 4] — the atom, again. Same object as the neuron's weighted sum (page 3) and page 27's similarity meter between king and queen. What is new: here the dot product is computed between every pair of tokens at once — the matrix $QK^\top$ is nothing but a table of dot products, one per query–key pair. Reopen: page 27, where you saw that a dot product's value depends on which directions you measure along. On this page the network gets to learn those directions — that is what $W_Q$ and $W_K$ are.
🔮 Predict first Self-attention has every token produce three things — a query, a key, and a value. Before you scroll: are those three different things the token stores (like three memory slots), or three views of one and the same vector? Commit to an answer.

The answer is three views of one vector, and the whole page turns on it. In code the resolution is a single line — q, k, v = x @ Wq.T, x @ Wk.T, x @ Wv.T — the same x, projected three times. Hold that; we will earn it below and box it as the misconception that, left uncorrected, makes cross-attention (and therefore diffusion conditioning) impossible to read.

Start from a lookup you already know

You have written this a thousand times:

Python
memory = {"cat": v_cat, "dog": v_dog, "car": v_car}
memory["cat"]     # -> v_cat
memory["kitten"]  # -> KeyError

A Python dict has exactly three kinds of object: the query you look up ("kitten"), the keys it is compared against ("cat", "dog", …), and the values returned (v_cat, …). It has two flaws that make it useless as a layer in a learning machine:

  • It is brittle. "kitten" is cat-ish, but exact-match equality returns nothing. We want graded relevance, not a yes/no.
  • It is not differentiable. Hard equality is a step function — its derivative is zero almost everywhere. No gradient means the network can never learn what to look up. This is the chain rule's veto: a link with zero local slope kills the whole product (page 5).

Soften both flaws with one move you have made before. Replace equality with a graded similarity score — the dot product, page 27's atom — and replace "return the one match" with a weighted average of every value, weighted by how well each key scored. For a single query $\mathbf{q}$ against keys $\mathbf{k}_j$ and values $\mathbf{v}_j$:

$$\text{output} \;=\; \sum_j \underbrace{\frac{\exp(\mathbf{q}\cdot\mathbf{k}_j)}{\sum_{j'}\exp(\mathbf{q}\cdot\mathbf{k}_{j'})}}_{A_{j}\;=\;\text{how much key }j\text{ matched}} \,\mathbf{v}_j$$

SymbolIsShapeFrom
$\mathbf{q}$the query — "what am I looking for?"$(d_{\text{head}},)$a projection of one token, below
$\mathbf{k}_j$key $j$ — "what does token $j$ advertise?"$(d_{\text{head}},)$a projection of token $j$
$\mathbf{v}_j$value $j$ — "what does token $j$ hand over if picked?"$(d_{\text{head}},)$a projection of token $j$
$\mathbf{q}\cdot\mathbf{k}_j$the raw match score (a dot product)scalarthe atom, page 27

🔑 The one sentence Attention is a dictionary lookup where the keys don't have to match exactly, and instead of one value you get a blend of all of them, weighted by how well each key matched. Softening the lookup is what makes it differentiable — and differentiable means the network can learn what to look for.

Tokens don't arrive with a query, key, and value — they compute them

The one thing missing: a raw token embedding $\mathbf{x}_i$ is not a query, or a key, or a value. Each of those is a learned projection of $\mathbf{x}_i$. Stack the whole sequence as rows of $X$ and project all tokens at once (row-batched, as all attention is):

$$Q = X W_Q^\top, \qquad K = X W_K^\top, \qquad V = X W_V^\top$$

shape ribbon
X            (S, d)                 # S tokens, each a d = d_model vector
W_Q, W_K     (d_head, d)            # (out, in) — ALWAYS
W_V          (d_head, d)            # (out, in)
Q = X @ W_Q.T                       (S, d_head)
K = X @ W_K.T                       (S, d_head)
V = X @ W_V.T                       (S, d_head)
🌐 Translation note Papers write $\mathbf{q} = W_Q\mathbf{x}$ with column vectors and $W_Q \in \mathbb{R}^{d\times d_{\text{head}}}$. We use the row-batched form for all attention (notation §1.3), so the weight is transposed in the matmul and its shape stays $(d_{\text{out}}, d_{\text{in}}) = (d_{\text{head}}, d)$ — the invariant never moves; only the batch placement does. Papers also write the per-head width as $d_k$, $d_v$, $d_h$, or head_dim; we write $d_{\text{head}}$. And position is $i$ (papers use $t$, $i$, $n$, or $j$; sequence length is $S$, papers' $T$, $L$, $N$, or seq_len).

The three roles, said plainly, are the entire intuition:

  • $\mathbf{q}_i = \mathbf{x}_i W_Q^\top$ — "what am I looking for?"
  • $\mathbf{k}_j = \mathbf{x}_j W_K^\top$ — "what do I advertise?"
  • $\mathbf{v}_j = \mathbf{x}_j W_V^\top$ — "what do I hand over if I'm picked?"

Why three projections and not one?

The natural objection: if a query and a key are both just a token, why not use one projection for both and save two-thirds of the weights? Two reasons, and each is load-bearing.

1. $Q \ne K$ buys asymmetric relevance. Take the sentence "the animal didn't cross the street because it was too tired." The token it needs to look hard at animal — but animal has little reason to look back at it. Relevance is directed. If $W_Q = W_K$ then every score is $S_{ij} = \mathbf{q}_i\cdot\mathbf{k}_j = (\mathbf{x}_i W_Q^\top)\cdot(\mathbf{x}_j W_Q^\top)$, which is symmetric in $i$ and $j$: $S_{ij} = S_{ji}$, forced. Directed relations are gone. Choosing different directions for the asker and the advertiser is exactly what breaks the symmetry — and it is the same fact you met on page 27, where king·queen depended on which directions you measured along. $W_Q$ and $W_K$ are that choice of directions, now learned, and chosen separately.

2. $V \ne K$ separates "how I'm found" from "what I contribute." A key is the token's advertisement; a value is its payload. Paris should advertise "ask me about France" (via $\mathbf{k}$) while carrying the geographic content it will actually hand over (via $\mathbf{v}$). The analogy is a database: the index you search is not the row it points at — you do not store the whole row inside the B-tree. Tying $V$ to $K$ would force the thing a token is found by to equal the thing it gives you, and those are genuinely different jobs.

⚠️ The #1 attention misconception "Q, K, V are three different things the model has — like three memories." They are not. They are three projections of the same vector. In self-attention every token makes all three from its own $\mathbf{x}_i$:
Python
q, k, v = x @ Wq.T, x @ Wk.T, x @ Wv.T   # same x, three times
This is the exact hinge for cross-attention (page 31): a learner who believes Q/K/V are three separate memories cannot understand how a query can come from an image while keys and values come from text — which is precisely how diffusion conditioning works. Fix the picture now and that page is free.

Scaled dot-product attention, every shape

Assemble the pieces. Every score at once is the matrix $QK^\top$; a softmax over each row turns scores into weights; those weights blend the values. In frozen notation:

📐 The formula $$\text{Attention}(Q,K,V) \;=\; \text{softmax}\!\left(\frac{QK^\top}{\sqrt{d_{\text{head}}}} + M\right)V$$
SymbolIsShapeFrom
$X$the input token sequence$(S, d)$the residual stream
$W_Q, W_K, W_V$the three projections$(d_{\text{head}}, d)$ — (out, in)learned parameters
$Q$queries, one row per token$(S, d_{\text{head}})$$X W_Q^\top$
$K$keys$(S_{\text{kv}}, d_{\text{head}})$$X W_K^\top$ (kv source)
$V$values$(S_{\text{kv}}, d_{\text{head}})$$X W_V^\top$ (kv source)
$QK^\top$the raw score table (dot products)$(S, S_{\text{kv}})$every query · every key

In production every attention runs with a batch axis and $H$ heads in parallel. The full shape ribbon:

shape ribbon · batched, with heads
Q          (B, H, S,     d_head)
K          (B, H, S_kv,  d_head)
scores = Q @ K.transpose(-1,-2)          (B, H, S, S_kv)     # / sqrt(d_head)
A = softmax(scores, dim=-1)              (B, H, S, S_kv)     # last axis = keys
O = A @ V                                (B, H, S, d_head)
📐 Three shape facts to hammer
  1. The output has $S$ rows — one per query. The key/value set $S_{\text{kv}}$ can be any length, independent of $S$. This single fact is what makes cross-attention obvious on page 31: let $K,V$ come from a different, shorter sequence and nothing else changes.
  2. Softmax is over the last axis — the keys. So row $i$ of $A$ sums to 1; columns do not. Reading a heatmap the other way (columns as probabilities) is the single most common misread of an attention map.
  3. The wiring is dimensionally rigid. $d_{\text{head}}$ links $Q$ and $K$; $S_{\text{kv}}$ links $K$ and $V$; $d = d_{\text{model}}$ never appears again after the projections. There is exactly one legal way to connect these tensors — [THREAD:Qwen3-8B beat 6] for Qwen3-8B that is $d_{\text{head}}=128$ and $H=32$ (constants.md §1.1), the same fourteen numbers we count all the way to the full parameter tally on page 36.

Why softmax specifically? Two reasons, both the "soften to differentiate" move made concrete. First, softmax forces $\sum_j A_{ij}=1$ with every $A_{ij}>0$, so the output is a convex combination of the value vectors — it lives inside their convex hull and cannot blow up; the layer is stable by construction. Second, it is smooth, so gradient flows to every key, including the ones that lost. A hard argmax would hand all the gradient to the single winner and none to the rest — nothing else could ever learn to compete. (The softmax's sharpness is set by the logit scale, which is exactly what $\sqrt{d_{\text{head}}}$ controls — the whole story of page 30.)

✍️ Worked example — one score, both directions Take $d = d_{\text{head}} = 2$ so we can do it by hand, with $$W_Q = \begin{bmatrix} 1 & 0.3 \\ -0.2 & 1 \end{bmatrix},\qquad W_K = \begin{bmatrix} 0.8 & -0.4 \\ 0.5 & 0.9 \end{bmatrix},$$ and two tokens $\mathbf{x}_{\text{it}} = (0.9,\,0.3)$, $\mathbf{x}_{\text{animal}} = (0.6,\,0.9)$. Project (row-batched $\mathbf{q} = \mathbf{x}W_Q^\top$, which equals the column form $W_Q\mathbf{x}$):
$\mathbf{q}_{\text{it}} = (1\cdot0.9 + 0.3\cdot0.3,\; -0.2\cdot0.9 + 1\cdot0.3) = (0.99,\,0.12)$
$\mathbf{k}_{\text{animal}} = (0.8\cdot0.6 - 0.4\cdot0.9,\; 0.5\cdot0.6 + 0.9\cdot0.9) = (0.12,\,1.11)$
The score it gives animal is the dot product: $$S_{\text{it},\,\text{animal}} = \mathbf{q}_{\text{it}}\cdot\mathbf{k}_{\text{animal}} = 0.99(0.12) + 0.12(1.11) = 0.1188 + 0.1332 = \mathbf{0.2520}.$$ Now the reverse direction. With $\mathbf{q}_{\text{animal}} = (0.87,\,0.78)$ and $\mathbf{k}_{\text{it}} = (0.60,\,0.72)$: $$S_{\text{animal},\,\text{it}} = 0.87(0.60) + 0.78(0.72) = 0.522 + 0.5616 = \mathbf{1.0836}.$$ $0.2520 \ne 1.0836$: the score matrix is asymmetric, exactly because $W_Q \ne W_K$. ⚠️ What rank collapse foreshadows When you press Collapse $W_K$ → rank 1, all six key arrows fall onto a single line and the score matrix's columns become proportional — every query gets nearly the same answer, and the layer can no longer distinguish keys. That failure has a name and a cure: it is a rank-deficient weight, and the machinery for measuring and exploiting the rank of a weight matrix is SVD and LoRA (Part IV, pages 39–40) — the exact tool you will use to fine-tune. Attention is also the first place you meet the idea that the rank of a projection is a real, visible thing.
Run this · code/attention_from_scratch.py The boxed formula on real tensor shapes, checked against PyTorch's fused kernel. The two asserts are the lesson: rows of $A$ sum to 1 (columns do not), and the hand-written version matches F.scaled_dot_product_attention to floating-point tolerance.
code/attention_from_scratch.py
import torch
import torch.nn.functional as F

torch.manual_seed(0)
S, S_kv, d_head = 4, 6, 8          # queries, keys/values, per-head width
Q = torch.randn(S,    d_head)
K = torch.randn(S_kv, d_head)
V = torch.randn(S_kv, d_head)

def attention(Q, K, V):
    scores = Q @ K.T / (d_head ** 0.5)   # (S, S_kv) — a table of dot products
    scores = scores - scores.max(dim=-1, keepdim=True).values  # online-safe softmax
    A = scores.exp()
    A = A / A.sum(dim=-1, keepdim=True)      # softmax over LAST axis = keys
    return A @ V, A                          # (S, d_head), (S, S_kv)

O, A = attention(Q, K, V)
print("Q", tuple(Q.shape), "K", tuple(K.shape), "scores/A", tuple(A.shape), "O", tuple(O.shape))

# Rows sum to 1; columns do NOT (notation §7).
assert torch.allclose(A.sum(dim=-1), torch.ones(S), atol=1e-6)
assert not torch.allclose(A.sum(dim=0), torch.ones(S_kv), atol=1e-3)

# Matches PyTorch's fused kernel (same 1/sqrt(d_head) scale).
O_ref = F.scaled_dot_product_attention(Q, K, V)
assert torch.allclose(O, O_ref, atol=1e-5)
print("ok: hand-written attention == F.scaled_dot_product_attention")
