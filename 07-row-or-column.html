Row or Column? The Shape Bridge

Page 6 handed you $\mathbf y = W\mathbf x$ on paper and a1 = tanh(lin1(x)) in PyTorch, and quietly let the question sit: are those the same $W$? They are. This page proves it, gives you the one rule that never changes, and closes the loop page 6 deliberately left open with nn.Linear(2, 2).

🔮 Predict first nn.Linear(3, 5) — what is .weight.shape? (a) (3, 5) or (b) (5, 3)? Commit to one, then read on — we'll see why the maths and the code agree even though one writes $W\mathbf x$ and the other writes $x A^\top$.

The ground truth

[VP] PyTorch 2.13. torch.nn.Linear applies $y = xA^\top + b$, and its weight has shape (out_features, in_features). Every maths treatment in this course writes $\mathbf y = W\mathbf x + \mathbf b$ with $\mathbf x$ a column. Put a real layer through both readings:

Python
layer = nn.Linear(3, 5)
print(layer.weight.shape)   # torch.Size([5, 3])

nn.Linear(3, 5).weight.shape == (5, 3) — answer (b). And the maths's $W$ in $\mathbf y = W\mathbf x$, for that same layer, is also $(5,3)$: five rows (one per output), three columns (one per input). There is no second matrix to reconcile. It is the identical $15$ numbers, read two ways.

🔑 The reconciling gift There is no disagreement about $W$. $\texttt{nn.Linear(3, 5).weight.shape} == (5, 3)$, and the maths's $W$ in $\mathbf y = W\mathbf x$ is also $(5,3)$. The two conventions differ only in whether you stack a batch on the left. Every other course you've half-read that made this feel confusing was fighting a battle that doesn't exist.
📐 The prescription
ContextConventionForm
Single-example maths; all trunk derivations; backpropColumn vectors $\mathbf y = W\mathbf x + \mathbf b$,   $\mathbf x \in \mathbb R^{d_{in}\times 1}$
Anything with a batch or sequence axis; all code; all attention Row-batched $Y = XW^\top + \mathbf b^\top$,   $X \in \mathbb R^{B\times d_{in}}$
Everywhere, no exceptions: $W$ is $(d_{\text{out}}, d_{\text{in}})$.
SymbolIsShapeFrom
$W$the layer's weight matrix$(d_{out}, d_{in})$learned parameters — the same matrix in both forms
$\mathbf x$one input example, a column$(d_{in}, 1)$single-example maths
$X$a batch of inputs, stacked as rows$(B, d_{in})$code / row-batched maths
$\mathbf b$bias$(d_{out}, 1)$learned parameters
$\mathbf y$output, single example$(d_{out}, 1)$column form: $\mathbf y = W\mathbf x + \mathbf b$
$Y$output, whole batch$(B, d_{out})$row form: $Y = XW^\top + \mathbf b^\top$

Why split rather than pick one globally. Column-only would force transposing every attention equation you'll ever read — attention is written row-style universally in the literature, and fighting that spends credibility to buy nothing. Row-only makes the backprop derivation on page 14 uglier: gradients become row covectors and the Jacobian bookkeeping that makes backprop click gets muddier right where it needs to be cleanest. The split costs exactly one page — this one. In exchange, the thing you must remember never changes: only the batch placement does, and the shape ribbon shows you that on every line.

The same object, both shapes

Page 6's TN-1 ran one example, column form. Here is that exact hidden layer, first as you saw it, then stretched to a batch — same $W_1$, same numbers:

Shape ribbon
x            (2,)                                # one example — page 6
W1           (2, 2)     b1  (2,)                 # (out, in) — ALWAYS
z1 = W1 x + b1                  (2,)              # column form

X            (B, 2)                              # a batch of B examples
W1           (2, 2)     b1  (2,)                 # (out, in) — SAME W1
Z1 = X @ W1.T + b1.T             (B, 2)            # row-batched form

Three facts to hammer here — they recur unchanged at attention (pages 29–30):

  • The output has one row per input example; the batch axis is purely a hardware convenience — each row is computed independently.
  • $d_{\text{in}}$ disappears after the projection; only $d_{\text{out}}$ survives into the next shape.
  • The operation is dimensionally rigid — there is exactly one legal wiring, and the shape ribbon makes a mis-wiring visible before you run anything.

And the anchor model makes the invariant concrete at real scale — [THREAD:Qwen3-8B beat 3]: Qwen3-8B's down_proj is nn.Linear(12288, 4096), so its weight is (4096, 12288) — (out, in), exactly as page 2's invariant demanded, at nine digits instead of two (constants.md §1.2).

⚠️ "The maths and the code use different weight matrices." Same $W$, transposed multiply. x @ W.T and W @ x (per example) are the same numbers — the shapes_bridge.py demo below asserts it bit-for-bit, not just approximately.
⚠️ "(2, 2) proves I understand the convention." It can't. When out == in, $(d_{out}, d_{in})$ and $(d_{in}, d_{out})$ are the same tuple — page 6's nn.Linear(2, 2) was deliberately ambiguous for exactly this reason. That's why every demo on this page uses nn.Linear(2, 3): with $(out,in)=(3,2)$, a transposed load is unmistakable.
⚠️ "Transpose moves data." In PyTorch, .T is a stride change — a new view over the same bytes, no copy — until a kernel demands .contiguous() and forces one. Cheap to reach for; not free to abuse in a hot loop.
🔬 Deep dive: what the backprop page would look like if the architect had gone row-only

Row-only would make every gradient a row covector instead of a column, because the forward pass is $Y = XW^\top$ and differentiating through a transpose flips which side every Jacobian multiplies on. Page 14's clean recursion $\boldsymbol\delta^{(\ell)} = (W^{(\ell+1)\top}\boldsymbol\delta^{(\ell+1)}) \odot \phi'(\mathbf z^{(\ell)})$ would instead read $\boldsymbol\delta^{(\ell)} = (\boldsymbol\delta^{(\ell+1)} W^{(\ell+1)}) \odot \phi'(\mathbf z^{(\ell)})$ with $\boldsymbol\delta$ a row vector at every layer — correct, but it obscures the "gradient has the same shape as the thing it's the gradient of" law that makes the recursion legible. That one clarity is the entire justification for column form in the trunk; it costs this one bridging page.

Verify it yourself

Try it code/shapes_bridge.py — builds nn.Linear(2, 3), prints weight.shape → (3, 2), then computes the same layer's output as $W\mathbf x$ (column) and as $x W^\top$ (row) and asserts torch.allclose. Runs on CPU in under a second.
Python · code/shapes_bridge.py
import torch

SEED = 42
torch.manual_seed(SEED)

# nn.Linear(in_features=2, out_features=3): weight is ALWAYS (out, in).
layer = torch.nn.Linear(2, 3)
print(tuple(layer.weight.shape))   # (3, 2)
assert tuple(layer.weight.shape) == (3, 2)

W = layer.weight.detach().clone()      # (3, 2) — the SAME matrix, both forms below
b = layer.bias.detach().clone()        # (3,)
x = torch.tensor([1.0, 2.0])           # one example, column-form shape (2,)

# Column form: single-example maths, y = W x + b.
y_col = W @ x + b                      # (3,2) @ (2,) -> (3,)

# Row-batched form: code's convention, Y = X W^T + b. Batch of 1, same x.
X = x.unsqueeze(0)                     # (1, 2) — x promoted to a one-row batch
y_row = X @ W.T + b                    # (1,2) @ (2,3) -> (1,3)

assert torch.allclose(y_col, y_row.squeeze(0))
print("y_col == y_row.squeeze(0):", torch.allclose(y_col, y_row.squeeze(0)))
