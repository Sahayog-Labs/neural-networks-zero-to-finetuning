Teaching Behaviour: SFT, Chat Templates & Preference Alignment

Page 47 decided whether to fine-tune. This page is what "fine-tune" actually means once you've decided: the same cross-entropy loss you already own, aimed at a chat-formatted conversation through a mask — and then, one rung up, taught to prefer one answer over another instead of just imitating a single correct one.

LLM Fine-Tuning Track

Three ideas live on this page, in the order a real pipeline hits them: SFT teaches imitation of a masked target, a chat template is the silent contract that makes the mask meaningful, and preference alignment (DPO, and its 2026 cousins) teaches preference instead of imitation. The page ends at the doorstep of GRPO — the capstone's engine.

🔮 Predict first Your SFT dataset has 400-token prompts and 50-token answers. If you compute the loss on every token — prompt included — what fraction of your gradient is spent teaching the model to write the user's questions rather than the assistant's answers?

1 · Post-training is behaviour, not knowledge

Hold onto this frame; every box on this page is a special case of it:

Post-training gives a pretrained model manners, a format, and the willingness to answer. It changes almost nothing about what the model knows and almost everything about what it does.

The pipeline you're about to walk, one stage at a time:

Base model —SFT→ instruction-following —Preference→ aligned —RLVR→ reasoning

Each arrow is a training run over a different loss on the same pretrained weights. None of them touches pretraining's next-token objective — they change what it's applied to and which positions count.

2 · SFT's loss — the same cross-entropy, plus a mask

[THREAD:chain rule beat: SFT loss] — trunk callback: this is not a new algorithm. It is the exact per-position cross-entropy you derived on the trunk's logs/softmax/cross-entropy page and met again as pretraining's objective on page 45, with one addition — a mask that zeroes out most of the sequence:

$$\mathcal{L}_{\text{SFT}}(\theta) \;=\; -\sum_{i=1}^{S} m_i \,\log p_\theta\!\big(y_i \mid y_{<i}\big), \qquad m_i \in \{0,1\}$$

SymbolIsShape / rangeFrom
$\mathcal{L}_{\text{SFT}}(\theta)$the masked SFT loss for one training examplescalar, natssame per-example CE, trunk-owned
$\theta$all trainable parameters (or, with LoRA, just $A,B$)$(P,)$notation §4.1
$S$sequence length of this rendered examplescalarnotation §3.2 — papers write $T,L,N$
$m_i$the loss mask at position $i$ — this page's new object$\{0,1\}$implemented as label = -100
$y_i$the true next token at position $i$integer in $[0,V)$the rendered, tokenized example
$y_{<i}$every token before position $i$prefixthe conditioning context
$p_\theta(\cdot \mid y_{<i})$the model's softmax over the vocabulary$(V,)$the exact object from page 45, $V=151{,}936$ classes

Translation note. Papers and library docs call $m_i$ the loss mask, the label mask, or simply "-100 labels" (Hugging Face's ignore_index convention — any position whose label is -100 contributes exactly zero to the sum). This course writes $m_i$ so it sits next to the sequence-position index $i$ you already own.

TRL 1.8's three masking regimes for a message-list dataset [VP] — the demo below lets you flip between all three live:

  • completion_only_loss=True — the default. Treats the conversation as one prompt plus one completion: only the final turn counts.
  • assistant_only_loss=True — needs {% generation %} / {% endgeneration %} markers in the template; masks every assistant turn, not just the last. TRL auto-patches Qwen3's template with these markers; check before assuming another model's template has them.
  • neither flag — full-sequence loss. Correct only for continued pretraining, wrong for chat SFT.

3 · Chat templates — the #1 silent failure

A mask only means something once the conversation has been turned into one flat token sequence with unambiguous turn boundaries, and that rendering step is a real contract, not a formatting nicety. Qwen uses ChatML: <|im_start|>role\ncontent<|im_end|>\n, repeated per turn.

📐 Two rules that are not optional
  • Always call tokenizer.apply_chat_template(). Never hand-roll the string. The template is versioned with the checkpoint; the library owns it so you don't have to guess it.
  • Train-time and inference-time rendering must match exactly — including the trailing <|im_start|>assistant\n that inference adds (add_generation_prompt=True) and training does not. The EOS you train on must equal the turn terminator: for Qwen, SFTConfig(eos_token="<|im_end|>").

The " the"-vs-"the" byte fact from page 44 is exactly why hand-rolling breaks silently: a template built by string-concatenation that puts a space in the wrong place produces a different, valid token sequence — no error, no exception, just a model quietly trained on input it will never see again at inference time.

⚠️ "Loss masking is a minor implementation detail" If you train on the user's own turns, you are teaching the model to generate plausible user questions — it will start interviewing you. On a dataset of 400-token prompts and 50-token answers, computing loss on every position spends 89% of your unmasked gradient on text the model should never generate. That number ends the argument.
✍️ Worked example — resolving the predict Prompt $=400$ tokens, answer $=50$ tokens, full-sequence loss (no mask): $$\frac{400}{400+50} = \frac{400}{450} = 0.8888\overline{8} \approx \mathbf{89\%}$$ Eighty-nine percent of the unmasked gradient pushes the model toward writing your questions, not its answers. 4 · How many examples does SFT actually need?

[THREAD:memory ledger beat: dataset size, not model size] — trunk callback to D-11/D-14's information-content rule, the same principle that sets LoRA rank on page 40: the right dataset size scales with how much information you're asking the model to acquire, not with how big the model is.

GoalExamples [EST, folklore/practitioner consensus]Note
Format / style only50–200the model already "knows how," it just needs the shape
Tone / persona200–1,000consistent voice across many prompts
Domain idiom / jargon1,000–10,000vocabulary and conventions, not new facts
New capability / skill10,000–100,000genuinely new behaviour, not a repackaging
Change what it knows$\infty$ — this is RAG's jobtrunk callback to page 47's misconception

LIMA's own result is the sharpest data point here: 1,000 curated examples matched or beat much larger instruction-tuned baselines — and scaling the same recipe to 2,000 examples without maintaining the same curation quality made the model worse. Low-quality data has negative value, not zero value; this is why "just add more examples" is not automatically the right move.

5 · Preference alignment — teaching "better," not just "correct"

SFT shows the network one right answer per prompt. Direct Preference Optimization (DPO) shows it a preferred and a rejected answer to the same prompt, and its key insight is that you don't need a separately trained reward model at all: the optimal KL-constrained policy has a closed form, so you can optimize the policy directly against preference pairs.

$$\mathcal{L}_{\text{DPO}} = -\,\mathbb{E}_{(x,y_w,y_l)}\!\left[\log\sigma\!\left(\beta_{\text{DPO}}\log\frac{\pi_\theta(y_w\mid x)}{\pi_{\text{ref}}(y_w\mid x)} - \beta_{\text{DPO}}\log\frac{\pi_\theta(y_l\mid x)}{\pi_{\text{ref}}(y_l\mid x)}\right)\right]$$

SymbolIsShape / rangeFrom
$\pi_\theta$the policy being trained — Qwen3-8B, or its LoRA$p(\text{completion}\mid x)$the SFT checkpoint, continuing to train
$\pi_{\text{ref}}$the frozen reference policy — the SFT model before this stagesamea fixed snapshot, never updated
$x$the shared prompttoken sequencethe dataset
$y_w, y_l$the preferred ("win") / rejected ("lose") completiontoken sequencesnotation §4.5, a human or program's preference judgement
$\beta_{\text{DPO}}$KL strength — how far $\theta$ may drift from $\pi_{\text{ref}}$scalar, $\approx 0.1$a training hyperparameter
$\sigma(\cdot)$logistic sigmoid, with an explicit argument — legal here (notation §5.2)$(0,1)$trunk-owned

[THREAD:chain rule beat: DPO gradient] — the gradient of $\mathcal{L}_{\text{DPO}}$ with respect to $\theta$ is the same chain rule you've used since page 5, run through a log-ratio-of-policies instead of a cross-entropy; $\sigma'(\cdot)$ is the same sigmoid derivative from the trunk's activation pages.

🔑 The LoRA trick that makes DPO nearly free [THREAD:memory ledger beat: a reference model for zero extra bytes] — reopens page 18/49's ledger. If $\pi_\theta$ is a LoRA adapter on top of $\pi_{\text{ref}}$'s frozen base, you get $\pi_{\text{ref}}$ for free by disabling the adapter — one model in VRAM, not two. No second 16.38 GB copy of Qwen3-8B's weights is needed just to hold a reference policy still.

🔬 Deep dive: why DPO's reward has a closed form at all

Start from the standard RLHF objective — maximize expected reward $r(x,y)$ subject to a KL penalty that keeps $\pi_\theta$ close to $\pi_{\text{ref}}$: $\max_\theta \mathbb{E}_{y\sim\pi_\theta}[r(x,y)] - \beta_{\text{DPO}}\,\mathrm{KL}(\pi_\theta \| \pi_{\text{ref}})$. This has a known closed-form optimal policy, $\pi_\theta(y\mid x) \propto \pi_{\text{ref}}(y\mid x)\exp\!\big(r(x,y)/\beta_{\text{DPO}}\big)$. Inverting that relation expresses the reward in terms of the policy itself: $r(x,y) = \beta_{\text{DPO}}\log\frac{\pi_\theta(y\mid x)}{\pi_{\text{ref}}(y\mid x)} + \text{const}$. Substitute that into a Bradley-Terry preference model (the probability that $y_w$ beats $y_l$ is $\sigma(r(x,y_w)-r(x,y_l))$) and the intractable partition function cancels exactly, leaving the loss printed above — a loss over $\pi_\theta$ alone, with no reward model ever trained.

⚠️ "Alignment teaches the model new facts" It doesn't. DPO, like SFT, reweights behaviours the base model already had — it shifts probability mass toward outputs a human or program preferred; it cannot install knowledge the base network never possessed. Attempting to teach new facts via preference pairs produces the exact same failure mode as page 47's SFT misconception: confident, fluent, wrong. Same mechanism, different loss.

6 · GRPO & RLVR — the on-ramp to the capstone

DPO needs a fixed pair per prompt, judged once. Group Relative Policy Optimization (GRPO) instead samples a group of $G$ completions for the same prompt, scores each with a reward, and turns the group's own statistics into an advantage — no learned value network required:

$$A_i = \frac{r_i - \mathrm{mean}(r_1,\dots,r_G)}{\mathrm{std}(r_1,\dots,r_G)}, \qquad G \approx 8\text{–}64$$

SymbolIsShape / rangeFrom
$A_i$the advantage assigned to sample $i$ — how much better than the group averagescalarcomputed here, feeds the policy-gradient update
$r_i$the reward for sample $i$'s completionscalara program or a reward model — see RLVR, below
$G$group size — completions sampled per prompt$8$–$64$a training hyperparameter
$\mathrm{mean}(\cdot), \mathrm{std}(\cdot)$over the group's $G$ rewardsscalarsthe group is the baseline
🔑 The critic was replaceable by a sample mean PPO needs a learned value network just to know "better than average." GRPO asks the same question 16 times and takes the mean of the answers — the critic turns out to be a statistic you can compute for free from the samples you were already generating.

RLVR (reinforcement learning from verifiable rewards) is what makes $r_i$ trustworthy without a learned reward model at all: if a program can check the answer — a unit test passes, a math grader confirms the number, a JSON payload validates against its schema — you have a ground-truth reward and can run real RL, not an approximation of human taste. His own domain probably has a checker of exactly this kind (does the config parse? does the generated SQL run? does the extracted field match the schema?) — which is exactly the on-ramp page 52 walks through for real.
