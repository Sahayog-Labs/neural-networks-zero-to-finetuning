Vectors, Matrices, Tensors — Shape Is the Skill

Notation starts here. Not because the symbols are hard, but because almost every real mistake you will make fine-tuning a model is a shape mistake — and shape mistakes are the ones Python doesn't always have the courtesy to crash on.

🔮 Predict first You add a bias of shape (4096,) to a batch of activations of shape (32, 128, 4096). Does it (a) error, (b) add the bias to every row correctly, or (c) run but silently produce a (32,128,4096,4096) monster? Commit to one before you keep reading.

Shapes, with units

The last neuron you dragged (page 1) took two numbers in and produced one number out — no notation needed for that. The moment there's more than one neuron, you need a name for "how many numbers, arranged how." That name is shape, and its vocabulary is short:

🔑 Key idea Shape bookkeeping is the #1 practical skill in this course — not calculus. Almost every fine-tuning failure you will personally hit, from a loss that trains to nonsense to a CUDA out-of-memory crash, is a shape error or a memory error, and both start life as a shape you didn't check.

The one rule: matrix multiplication

Everything in a forward pass is built from one operation, applied over and over:

$$(AB)_{ij} = \sum_{k=1}^{n} A_{ik}B_{kj}, \qquad A\in\mathbb{R}^{m\times n},\ B\in\mathbb{R}^{n\times p} \ \Rightarrow\ AB\in\mathbb{R}^{m\times p}.$$

SymbolIsShapeFrom
$A$left operand$(m, n)$given
$B$right operand$(n, p)$given
$n$the inner dimensionscalarannihilates — it must match, and it vanishes from the output
$AB$the product$(m, p)$computed above

Play the cancellation game out loud: (m, [n) @ (n], p) → (m, p) — the two inner n's have to match, and once they've matched, they disappear. Every shape bug in this course is that game played wrong. The FLOP cost of one matmul is $2mnp$ (one multiply and one add per term, counted separately — the vendor convention this course uses everywhere); you won't need that number until the roofline pages (page 43), but it's cheap to bank now.

📐 The invariant $W$ is always $(d_{\text{out}}, d_{\text{in}})$. Always. In the maths, in the code, in every convention this course ever uses. Rows of $W$ are output neurons; columns are inputs. Page 7 is the whole page built to pay this sentence off — hold onto it until then.

Every multi-step tensor computation in this course carries its shapes inline, in this fixed format — the shape ribbon. Here is the canonical one, shown once and reused, never redrawn:

Shape ribbon
x        (B=32, S=128, d_model=4096)
W        (12288, 4096)        # (out, in) — ALWAYS
h = x @ W.T                    (32, 128, 12288)

Read it left to right: the input tensor, its shape with every axis named (not just sized); the weight, shape-and-role labelled; the operation, with the result shape it must produce. You'll see this exact strip under every tensor operation from here on — same format, same discipline, no exceptions.

Carried to a number: Qwen3-8B's FFN block

[THREAD:Qwen3-8B beat 1] — the anchor model's first arithmetic. Qwen3-8B is the model this whole course fine-tunes; from here on, its numbers recur until they're reflexive. One transformer block's FFN (SwiGLU) has three weight matrices, all built from $d_{\text{model}}=4096$ and $d_{\text{ff}}=12288$ ($=3d$) — and every one of them, correctly, is $(d_{\text{out}}, d_{\text{in}})$:

✍️ Worked example — the FFN's three matrices
$W_{\text{gate}}$maps $d_{\text{model}}\to d_{\text{ff}}$shape $(12288,\,4096)$$12288\times4096=\mathbf{50{,}331{,}648}$
$W_{\text{up}}$maps $d_{\text{model}}\to d_{\text{ff}}$shape $(12288,\,4096)$$\mathbf{50{,}331{,}648}$
$W_{\text{down}}$maps $d_{\text{ff}}\to d_{\text{model}}$shape $(4096,\,12288)$$\mathbf{50{,}331{,}648}$

Notice the invariant holds for every row of that table, even though $W_{\text{down}}$'s mapping runs the opposite direction from the other two — $(d_{\text{out}}, d_{\text{in}})$ doesn't care which way information flows, only which axis is which.

FFN per block $= 3\,d_{\text{model}}\,d_{\text{ff}} = 3\times4096\times12288 = \mathbf{150{,}994{,}944}$, which is 78.26% of the block's $192{,}946{,}432$ parameters (constants.md §1.2–1.3). Across all 36 blocks that's $150{,}994{,}944\times36 = 5{,}435{,}817{,}984$, and $5{,}435{,}817{,}984 / 8{,}190{,}735{,}360 = \mathbf{66.4\%}$ of the whole model (constants.md §1.3).

⚠️ Two different denominators — say which 78.26% and 66.4% are both true FFN shares — they just divide by different things (a block's params vs. the whole model's). Never say "the FFN is ~70% of the model" without naming which: that retired 70% figure was a Llama-3.1-8B number ($d_{\text{ff}}=3.5d$); Qwen3-8B's FFN is $d_{\text{ff}}=3d$, giving 66.4% of the model or 78.26% of a block — never 70% (constants.md §1.3, D-01).

[THREAD:memory beat 1] One line, then a deliberate pause: at bf16, those 8,190,735,360 parameters weigh 16.38 GB (decimal — constants.md §1.1). Weights are the cheap line item. Page 18 adds gradients and optimizer state on top and the number stops being something you can shrug at.

Broadcasting — and the bug that never raises

Shapes don't have to match exactly to combine — they have to be compatible. Align two shapes from the right; a pair of dims is compatible if they're equal, if either is $1$ (it gets stretched), or if one tensor simply runs out of dims (missing dims count as $1$). This is what lets a $(4096,)$ bias add cleanly to a $(32, 128, 4096)$ batch — the last axis matches, the rest just get copied down every row. (That resolves the predict above: it's (b) — add correctly, every row. The $(32,128,4096,4096)$ monster in option (c) is what happens when the bias picks up an accidental trailing axis, not when you add it directly — which is exactly the trap below, at a smaller scale.)

⚠️ The single highest-value warning in this chapter

(64,) is not (64, 1) is not (1, 64) — three different tensors, and PyTorch will let all three of them subtract from each other without complaint:

Python
pred   = torch.randn(64)      # (64,)
target = torch.randn(64, 1)   # (64, 1)
loss = ((pred - target)**2).mean()   # NO ERROR. Broadcasts to (64,64). Garbage.

Right-aligned, (64,) against (64,1): the last axes are $64$ vs.\ $1$ — compatible, the $1$ stretches. The leading $64$ on target has nothing across from it on pred, so pred is treated as if it had a leading $1$ there too — which then stretches to $64$ as well. Result: a $(64,64)$ matrix of every pred[j] - target[i], averaged into one number that looks like a loss and is not one. It runs. It trains. It converges — to nonsense. Habit: assert pred.shape == target.shape before every loss, no exceptions.

⚠️ $A \odot B$ is not $AB$ Elementwise (Hadamard) product $\odot$ multiplies same-shaped tensors entry-by-entry; matrix product $AB$ (`@`) contracts a shared inner dimension. They look similar on the page and do completely different things — SwiGLU (page 11) uses both, in the same line, on purpose: `down_proj(gate_act ⊙ up)` is a Hadamard product feeding a matmul. Confusing the two is a shape-passing bug, not a subtle one.

One more habit worth stating plainly: the batch axis is a hardware convenience, not a mathematical one. $(B, d)$ stacks $B$ independent examples so the GPU can matmul them together; mathematically, each row is its own, unrelated forward pass. Nothing in the maths above needed a batch axis — page 7 is where it gets added to the column-vector convention without changing $W$'s shape at all.
