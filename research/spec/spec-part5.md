# 44 — tokenization.html
**Title:** "Tokenization — the model never sees letters" · **Part V — Track A: LLMs** · build: **S** [recommended] · class `.track-llm`

**Objectives (4):** (1) Explain that text becomes integer IDs from a fixed ~152k dictionary learned by greedy compression; (2) run BPE mentally (count pairs → merge → replay merge-list); (3) compute token counts from the ρ≈4 rule and read $V$'s role in the loss's output dimension; (4) predict the three famous failure modes (miscounting letters, arithmetic, 2–3× cost on non-English) from the mechanism.

**PREDICT (opens page):** "Type `strawberry`. How many pieces do you think Qwen3's tokenizer splits it into — 1, 3, or 10? And why can a model that writes flawless essays not count the r's?"

**Sections:**
- *Intuition.* The one-sentence frame (brief §2): "The model sees integers drawn from a fixed dictionary of ~150,000 word-fragments, itself learned by a greedy compression algorithm from uncurated text." $V=151{,}936$ [VP, constants §1.1].
- *BPE mechanism.* $\tau:\Sigma^\*\to\{0,\dots,V-1\}^\*$, $|\Sigma|=256$ bytes. The 4-step loop (count pairs → merge argmax → append to ordered merge-list → repeat to $|V|$). **The under-taught point:** encoding is a *replay of the training-time greedy schedule*, not a search for optimal segmentation. Symbol Ledger for $\tau,\Sigma,V$.
- *Compression ratio.* $\rho=\text{chars}/\text{tokens}$; **$\rho\approx4$** for English prose, **1.5–2.5** for code/non-Latin [EST, constants §9.5]. Worked: "a 500-word doc ≈ 650 tokens." Byte-level BPE (Qwen/GPT lineage) vs SentencePiece: byte-level **can never emit `<UNK>`**; leading space lives *inside* the token (`" the"` ≠ `"the"`) — "this single fact causes a large fraction of chat-template bugs" (foreshadows p.48).
- *Why $V$ matters downstream.* Embedding + `lm_head` are each $V\times d = 151{,}936\times4096 = 622{,}329{,}856$ params [DER, constants §1.2]; together **15.20% of P** [DER, §1.3]. The loss on p.45 is cross-entropy over exactly these $V$ classes.
- **`.box warn` "Tokens are words":** they are word-*fragments*; whitespace and case make different tokens.
- **`.box warn` "I can just add tokens to the vocab":** new rows are randomly-initialized and untrained; naive vocab-extension degrades the model until you train the new embeddings.
- *Translation note:* we write $V$ for vocab; papers/config write `vocab_size`, sometimes $K$ (notation §3.2).

**Demo — real BPE trainer + tokenizer** (course.js `makeCtrl`, `Plot`; viz.js `hl` optional).
- Top panel: a text box; the learner's live text rendered with each token in an alternating background, token ID beneath.
- Bottom: `Plot` line chart, x = merges 0→3000, y = tokens-per-100-chars, **three traces: English / Python / Devanagari**.
- Sliders: `merges` (0→3000, step 50); `corpus` (English-only / mixed, 2-step).
- **Exact math the JS implements:** *real BPE training* over a bundled ~50 KB corpus — `Map<pair,count>`, each step finds argmax pair, applies the merge to word-splits, pushes to an ordered list; then encodes the live text by replaying the list in order (~80 lines; a lookup table is worthless).
- **Aha:** Devanagari's curve sits 2–3× above English at every merge count — "non-English costs more" becomes a *measured* line, not a claim.

**`.box try`** → `code/01_tokenizer_lab.ipynb`: load the real Qwen3 tokenizer, reproduce the 3-token `strawberry`, measure ρ on *his own* corpus (English vs code vs his jargon), and **print his corpus's total token count** — this number feeds p.49's wall-clock estimate. (~2 min.)

# 45 — pretraining-base-instruct.html
**Title:** "Next-token prediction, perplexity, and the two models you'll meet" · **Part V** · build: **S** · class `.track-llm`

**Objectives (4):** (1) State pretraining's objective as mean cross-entropy over positions and connect it to the trunk's CE-from-MLE; (2) compute step-0 loss and perplexity for Qwen3-8B; (3) distinguish **base** vs **instruct** and predict base-model behaviour ("it won't stop"); (4) explain why loss is comparable only within one tokenizer.

**PREDICT:** "Before any training, a freshly-initialized Qwen3-8B is asked to predict the next token. What loss (in nats) should it print on step 0 — and what does that number equal exactly?"

**Sections:**
- *Intuition.* Pretraining installs *everything the model knows*; it is next-token prediction over ~10^13 tokens.
- *The objective (frozen notation).* Per-position CE, batch-averaged:
  $$J(\theta)=-\frac1{B S}\sum_{b=1}^{B}\sum_{i=1}^{S}\log p_\theta\!\big(x_i \mid x_{<i}\big)$$
  Symbol Ledger ($J$ batch-avg loss in nats; $x_i$ token at position $i$; $x_{<i}$ prefix; $p_\theta$ softmax over $V$). **Trunk callback (`.track-banner`):** "this is Branch C of the MLE derivation you already did — cross-entropy per position. Not a new fact." (D-14).
- *Perplexity.* $\text{PPL}=e^{\mathcal L}$. **Step-0 sanity check:** loss $=\ln V=\ln(151{,}936)=\mathbf{11.93}$ nats; $\text{PPL}=V=151{,}936$ exactly, by construction [DER, constants §9.1]. **`.box warn`: NOT 11.76** — that is Llama-3's $V=128{,}256$, retired (D-01). Plot $\ln V$ as a horizontal "random-guess" line on every future loss curve. Well-trained 8B on English ≈ $\mathcal L\approx2.0$, PPL≈7.4 **[EST — corpus/tokenizer-dependent]**.
- *Base vs instruct.* Table: base = raw next-token continuer, usually **no chat template**, "won't stop"; instruct = SFT'd, has a mandatory chat template, answers instead of continuing. The instruct model is p.48's product.
- **`.box warn` "loss is comparable across models":** perplexity is only comparable within an identical tokenizer — different $V$/ρ ⇒ different nats/token. People compare cross-tokenizer PPL constantly; it is meaningless.
- *Translation note:* $i$ = sequence position here; papers write $t$ (notation §3.2).

**Demo — base vs instruct, live** (course.js `Plot`; ship pre-recorded real Qwen3 logits).
- Two columns: same prompt into `Qwen3-8B-Base` vs `Qwen3-8B`. Show token-by-token top-5 probabilities as small bars; a running generated string per side.
- Slider: `max_new_tokens` (1→60).
- **Exact math:** greedy/argmax decode from shipped real logit sequences; display cumulative log-prob. **Aha:** the base model keeps generating past a natural stop (no EOS habit); apply a chat template to the base and it *partially* recovers — "instruct-ness is a learned behaviour, not new knowledge" (sets up p.48's misconception).

**`.box try`** → `code/03_base_vs_instruct.py`: same prompt into base and instruct, print token-level probabilities, watch the base not stop, then wrap the base in a chat template. (~10 min.)

**Quiz (6):** (1) **Numeric** — step-0 loss for Qwen3-8B (num **11.93**, tol 0.05; distractor 11.76 = wrong-anchor). (2) Numeric — PPL at init (num **151936**, tol 0). (3) MC — SFT changes mostly *what the model does*, not *what it knows* (distractor: "teaches new facts" → the p.48/p.47 misconception). (4) MC — base model's defining behaviour (won't stop / no chat habit). (5) MC — why cross-model perplexity comparison is invalid (different tokenizer). (6) MC — $\ln V$ is the loss of (a uniform random guesser).

# 46 — decoding-kv-roofline.html
**Title:** "Getting text back out: decoding, the KV cache, and the decode roofline" · **Part V** · build: **O** [recommended — demo-heavy, load-bearing] · class `.track-llm`

**Objectives (5):** (1) Implement greedy/temperature/top-k/top-p/min-p from logits; (2) show temperature *never reorders* tokens; (3) derive the KV-cache bytes formula and compute 144 KiB/token for Qwen3-8B, with the GQA 4× win; (4) distinguish prefill (compute-bound) from decode (bandwidth-bound); (5) **apply the trunk roofline to LLM decode and use the decode heuristic to predict his box's tok/s.**

**PREDICT:** "The same Qwen3-8B on the same Spark. Prefill runs at ~8,000 tok/s. Decode (one token at a time) runs at ~20 tok/s. That's a ~390× gap on identical hardware. Guess why — and guess how many tokens/second bf16 Qwen3-8B decode will hit on *your* box."

**Sections:**
- *Decoding math (frozen).* Temperature: $p_v=\dfrac{\exp(z_v/T_{\text{samp}})}{\sum_{v'}\exp(z_{v'}/T_{\text{samp}})}$; $T_{\text{samp}}\to0^+$ ⇒ argmax (greedy), $=1$ ⇒ calibrated, $\to\infty$ ⇒ uniform. Top-k (keep k largest, $-\infty$ rest, softmax); top-p/nucleus (smallest set with cumulative $p\ge p_{\text{thresh}}$); min-p ($p_v\ge p_{\min}\cdot\max_{v'}p_{v'}$). **Translation note:** we write $T_{\text{samp}}$; papers/APIs write `temperature`/$T$ — and $T$ in this course is the diffusion chain length (notation §4.5, §3.2). Symbol Ledger on the temperature eq.
- **`.box warn` "temperature adds randomness":** it is a *monotone rescale of logits* — never changes the ranking, only the gaps. The demo proves it: greedy top token is identical at $T_{\text{samp}}=0.1$ and $2.0$.
- **`.box warn` "higher temperature = more creative":** the tail is mostly *wrong* tokens; beyond ≈1.2 you get degradation, not creativity.
- **`.box warn` "beam search is better":** it maximizes likelihood ⇒ bland/degenerate for open-ended chat ("likelihood trap"); use it only for single-correct-answer tasks.
- *The KV cache (frozen formula, constants §4):*
  $$\text{KV bytes}=2\cdot L\cdot H_{kv}\cdot d_{\text{head}}\cdot b\cdot S\cdot B$$
  **Note $H_{kv}$, not $H$** — the single most common field error; do not regress it. Symbol Ledger. Worked, Qwen3-8B ($L{=}36,H_{kv}{=}8,d_{\text{head}}{=}128,b{=}2$): per token $2\times36\times8\times128\times2=147{,}456$ B $=\mathbf{144\ KiB/token}$ [DER]. At $S{=}32{,}768$: **4.50 GiB**; at 40,960: **5.625 GiB**; at 131,072: **18.0 GiB > the 16.38 GB weights — the cache exceeds the model.** **GQA punchline:** MHA ($H_{kv}{=}32$) would be 576 KiB/token = **22.50 GiB** at 40,960; **GQA saves 4×** (constants §4). Why K and V but not Q: $q_i$ is used once; $K_{1:i},V_{1:i}$ are reused forever.
- **`.box warn` "the KV cache is optional if I'm short on RAM":** turning it off makes decode $O(S^2)$ → ~500× more compute; real levers are shorter context, smaller batch, fp8 KV.
- *Prefill vs decode.* Opposite bottlenecks. **Trunk-roofline callback (`.track-banner`, D-15):** arithmetic intensity $I=S_{\text{fwd}}$ FLOP/byte (constants §6.5). LLM decode $S_{\text{fwd}}=1\Rightarrow I=1$ — **227× below the working ridge of 227** (constants §6.4; ridge is **[INF]**, and 458 is the live alternative if the BF16 ceiling is ~125 TF — print the label, not a fact). Decode is bandwidth-bound; prefill ($S{=}2048\Rightarrow I{=}2048$) is compute-bound.
- ***The decode heuristic — the repaired version (constants §6.6, the byte-count is the lesson):***
  $$\boxed{\ \text{decode tok/s}\approx0.65\times\frac{\text{BW (GB/s)}}{\text{weight bytes read per token (GB)}}\ }\quad\text{dense models, batch 1}$$
  **[EST], ±10–15%.** The physically-correct byte count is **decode-traffic = checkpoint − embedding table** (the embedding is *gathered*, one row/token, not streamed). Under it, four independent dense batch-1 measurements land **60–75% MBU** (constants §6.7): Llama-3.1-8B FP8 60.3%, 70B FP8 69.8%, 8B NVFP4 70.5%, 14B NVFP4 74.7%. **The punchline `.box key`:** *"If your heuristic predicts more than 100% of peak bandwidth, your model is an MoE"* — gpt-oss-20B's implied 637 GB/s = 234% of the 273 GB/s bus is impossible because ~3.6B of ~20.9B params are active (constants §6.6). Second failure mode: DeepSeek-R1-14B at 83.5 tok/s is **batch 8, not 1**. **Qwen3-8B decode predictions [DER, constants §6.7]:** bf16 ceiling **16.67 tok/s**, expect **~10.8** (×0.65); fp8 33.33/~21.7; NVFP4 66.6/~43. Say which is ceiling and which is expectation.
- *Measured, on his box (constants §6.7) [VP]:* Llama-3.1-8B decode 20.5 (b=1) → **368 (b=32) = 18×**, while prefill is flat 7,991→7,949 — "the roofline, measured: decode was starving, prefill was already working." 273 GB/s [VP].

# 47 — prompt-rag-finetune.html
**Title:** "The decision: prompt vs RAG vs fine-tune — FORM vs FACTS" · **Part V — the decision page** · build: **O** · class `.track-llm`

**Objectives (4):** (1) Apply the one-sentence rule *fine-tuning teaches FORM, RAG supplies FACTS*; (2) walk the 7-question decision framework and stop at the first "yes," including the null action; (3) name and mechanically explain the field's most expensive mistake; (4) build a minimum-viable RAG pipeline (chunk→embed→retrieve→fuse→rerank→stuff) and know when a NumPy dot product beats a vector DB.

**PREDICT:** "You want your model to answer questions about your own 200 documents. Do you (a) fine-tune it on the documents, (b) put a search engine in front of it, or (c) improve the prompt? Commit to one before reading — most people pick (a), and it is usually the wrong and most expensive answer."

**Sections:**
- **The rule, biggest headline on the page:** *"Fine-tuning teaches FORM. RAG supplies FACTS. If you can't answer by pasting the answer into the prompt, fine-tuning won't fix it."* (brief §6). **Trunk callback (`.track-banner`, D-18):** this is the WHERE/WHAT axis of the "fine-tuning names three things" type-signature the trunk taught — FACTS ⇒ you change the *input*, not $\theta$.
- *The honest framework (7 questions, stop at first yes).* Table verbatim from brief §6: (1) better prompt / few-shot — works ~40% of the time, zero cost; (2–4) doesn't-know / needs-citation / knowledge-changes ⇒ **RAG**; (5) form/style/schema/idiom ⇒ **fine-tune**; (6) latency/cost of a 4,000-token prompt ⇒ **fine-tune (distill the prompt)**; (7) RAG works but the model won't use context ⇒ **both**.
- **`.box warn` — the biggest warning box in the course:** *"I'll fine-tune on my company docs so it knows them."* **Mechanism:** SFT on documents optimizes $-\log p_\theta(\text{doc tokens})$ → makes the model *write text that sounds like your docs*, not a retrievable index. Facts smear across 8.19B params at LR $2\times10^{-4}$; result = fluent, confident, correctly-formatted **wrong** answers — strictly worse, because the style now signals unearned authority. **Fix sentence:** *"Fine-tuning changes the model's prior, not its evidence. If you want a fact at inference time, the fact must be in the context window at inference time. There is no third option."*

**PREDICT:** "DeepSeek-R1 and the 2026 reasoning models learned to reason without a human labelling a single answer. What did they use instead of human preferences — and could you run a tiny version of it on your Spark tonight?"

**Sections:**
- *Intuition (brief §5).* SFT teaches imitation; DPO teaches preference; **RLVR teaches reasoning where you can check the answer with a program** (unit test, math grader, JSON-schema validator). "This is where reasoning models come from and it's the most *actionable* alignment idea, because his own domain probably has a checker" (does the config parse? does the SQL run? does the extracted JSON match the schema?).
- *GRPO math (frozen notation).* Sample a group of $G$ responses to a prompt; the advantage of response $i$ is
  $$A_i=\frac{r_i-\text{mean}(r_1,\dots,r_G)}{\text{std}(r_1,\dots,r_G)}$$
  Symbol Ledger ($r_i$ = scalar reward from the verifier for response $i$; $G$ = group size, typically **8–64**, TRL `num_generations` default 8–16 [VP]; mean/std over the group = the baseline). **The insight, ship it:** *"PPO needs a learned value network to know 'was this better than average?' GRPO just asks $G$ times and takes the mean. The critic was replaceable by a sample mean — that is why GRPO took over."* **Trunk callback (`.track-banner`, `[THREAD:chain rule]`):** the policy-gradient update is the chain rule once more — $\nabla_\theta$ of $\log\pi_\theta(\text{response})$ weighted by $A_i$; same verb, new loss (returns from p.48).
- *Why this needs the container (constants §6.9, D-10j, D-21).* GRPO is **online** RL — it must *generate* in the training loop (`--vllm_mode colocate`), which is slow and is exactly what gated this capstone until vLLM was unblocked. **The validated path:** the **`vllm/vllm-openai:cu130-nightly` container, pinned by digest** — the PyPI aarch64 wheel is CUDA-12 and fails on CUDA-13-only DGX OS. bitsandbytes' native sm_121 (constants §6.9) makes the quantized path viable too. TRL v0→v1: `GRPOConfig` `vllm_mode` default flipped `"server"`→`"colocate"` (constants §7.2) — a small break; bump the pin.
- *Runnable scope (D-21, D-01).* Generation-in-the-loop is expensive, so the capstone runs on **Qwen3-0.6B** (the smoke-test model), with the arithmetic transferring to the 8B anchor by substituting four config numbers ($d,L,H_{kv},d_{\text{ff}}$). Reward = a real `trl.rewards`-style verifiable function (e.g. `reasoning_accuracy_reward` or a schema validator he writes). Show a group of $G$ sampled completions, their rewards, the resulting $A_i$, and the policy update — "the model gets better at a task you can *check*, with no human labels and no reward model."
- **`.box warn` "GRPO is magic / RLVR installs knowledge":** like all post-training (p.48/p.47), it *reweights behaviours the base already has* toward verifiably-correct ones; it does not install facts. It works only where a program can score the output.
