Embeddings and the Dot Product

An address, and a meter that reads it

An embedding is a learned address — a point in a few-thousand-dimensional space. Once a token has been looked up, its identity as a token is gone; all that's left is its coordinates. Everything downstream — attention, the residual stream, the final softmax — only ever asks questions about those coordinates. The dot product is the question: "how much of this vector points along that one?" — a similarity meter with a needle. Positive: aligned. Zero: unrelated. Negative: opposed. You already use this instrument every time you read a cosine-similarity score off an embedding search; this page is where it gets a hand.

The math

Token lookup is a matmul in disguise: represent token $v$ as a one-hot row $\mathbf{o}_v \in \{0,1\}^V$, and the embedding is $\mathbf{e}_v = E^\top\mathbf{o}_v$ — literally selecting row $v$ of $E$. Writing it as a matmul (rather than "array indexing") is not pedantry: it is the reason gradients flow back into the embedding table at all — nn.Embedding's backward pass is exactly this matmul's backward pass, accumulated over every position that used token $v$ in the batch.

shape ribbon (embedding lookup): o_v (V,) one-hot at index v E (V, d) the whole table — (out, in) reads oddly here; E is a LOOKUP TABLE, not a Linear e_v = E.T @ o_v (d,) = row v of E, selected

Qwen3-8B [VP, constants §1.1]: $V = 151{,}936$, $d = 4096$, so $E \in \mathbb{R}^{151{,}936\times4096}$ holds

$$P_E = 151{,}936 \times 4096 = \mathbf{622{,}329{,}856} \approx 622\text{M}$$

— 7.6% of the model's 8,190,735,360 total parameters [VP, constants §1.1], sitting in a pure lookup table with no computation attached. [THREAD:Qwen3-8B] this number recurs whole on page 36's parameter count.]

Now the meter itself. For vectors $\mathbf{a},\mathbf{b}$:

$$\mathbf{a}\cdot\mathbf{b} = \sum_i a_i b_i = \|\mathbf{a}\|\,\|\mathbf{b}\|\,\cos\vartheta, \qquad \cos\vartheta \in [-1, 1]$$

Then attention's whole score matrix is nothing more than this meter, run in bulk. (Notice the overload: $S$ names the score matrix here and the sequence length in the shape ribbon below — the field does this too; context always disambiguates, since one is a matrix and the other a scalar.)

$$S = QK^\top, \qquad S_{ij} = \mathbf{q}_i\cdot\mathbf{k}_j$$

Say it plainly: $QK^\top$ is not mysterious. It is $S\times S$ dot products computed at once, because that is what a matmul is. Page 29 opens exactly here.

A worked example — three toy embeddings

✍️ Worked example Toy 3-D embeddings (these three vectors recur in page 29's attention widget): $\mathbf{v}_{\text{king}} = (0.9, 0.8, 0.1)$, $\mathbf{v}_{\text{queen}} = (0.85, -0.7, 0.15)$, $\mathbf{v}_{\text{bicycle}} = (-0.4, 0.05, 0.9)$.

king · queen $= (0.9)(0.85) + (0.8)(-0.7) + (0.1)(0.15) = 0.765 - 0.560 + 0.015 = \mathbf{0.220}$.
$\|\text{king}\| = \sqrt{1.46} = 1.208$, $\|\text{queen}\| = \sqrt{1.235} = 1.111$, so $\cos\vartheta = 0.220 / 1.342 = \mathbf{0.164} \Rightarrow \vartheta = 80.6°$.

king · bicycle $= -0.360 + 0.040 + 0.090 = \mathbf{-0.230}$, $\|\text{bicycle}\| = \sqrt{0.9725}=0.986$, $\cos\vartheta = -0.230/1.192 = \mathbf{-0.193} \Rightarrow \vartheta = 101.1°$.

The lesson: king and queen are nearly orthogonal here — 0.164 is barely off zero — because this toy space disagrees hard on the second axis (0.8 vs $-0.7$). Similarity is a projection; which directions you weight decides the answer. Plant the seed: $Q$ and $K$ are learned projections that choose exactly those directions (harvested on pages 29–30).

Now scale king by 10: the raw dot goes to $10\times0.220=\mathbf{2.20}$ — but $\cos\vartheta$ stays 0.164, unchanged. The raw dot product conflates direction (meaning) with magnitude (confidence). That conflation is precisely why 2026-era models add a normalization step to $Q$ and $K$ before scoring — Qwen3 ships q_norm/k_norm [VP] for this exact reason (pages 30/34).

What embedding space actually looks like

Near-orthogonality is the quantitative core of a bigger, honest picture — worth stating plainly before the deep dive so it isn't mistaken for decoration: raw cosine similarity is a poorly calibrated ruler on real contextual embeddings, and that miscalibration is a live bug source, not a curiosity.

🔬 Deep dive: anisotropy, analogies, and why 4096 dimensions hold more than 4096 concepts

(1) Anisotropy [M] — real contextual embeddings don't spread uniformly over the sphere; they occupy a narrow cone. Random-pair cosine among them runs high (~0.5–0.9), not near zero the way this page's random-unit-vector demo shows. This is the reason production RAG systems use contrastively-trained embedders rather than raw LLM hidden states — a raw hidden-state space isn't calibrated the way this page's toy vectors are (forward-ref RAG, Part IV).

(2) The king − man + woman ≈ queen analogy [M] is real but oversold — it only works reliably once you exclude the input words themselves from the nearest-neighbor search. Treat it as a suggestive party trick, not a load-bearing property.

(3) The linear-representation hypothesis [D] — features as directions in activation space, not individual coordinates. This is the implicit justification for why a low-rank additive update (LoRA, forward-ref) can steer behavior at all: if a concept is one direction, a rank-1 nudge along it is enough.

(4) Near-orthogonality, quantitatively — in $d=4096$, $\mathbb{E}[\cos\vartheta]=0$ with std $\approx 1/\sqrt{d} = 0.0156$; exponentially many almost-orthogonal directions fit in a space of only thousands of dimensions (the Johnson–Lindenstrauss lemma [VP] is solid; calling this "superposition" is a framing [M], not a theorem). This is the actual storage-capacity argument for why $d=4096$ can encode vastly more than 4096 independent "concepts."

Three traps

⚠️ "Each embedding dimension means something." Meaning lives in directions, generally not axis-aligned ones, and the linear-representation hypothesis implies there are more meaningful directions than there are dimensions to hold them one-per-axis. The Johnson–Lindenstrauss count above is the fix for this intuition, not a hand-wave.
⚠️ "Token X always has vector V." Only true of layer 0, the raw input lookup. One transformer block later the vector is contextual — "bank" in "river bank" and "bank account" diverge immediately, because attention has already mixed in neighboring tokens. The table $E$ is an address book, not the final answer.
⚠️ "Higher cosine means more related." — this one actively breaks RAG systems. Only inside a calibrated space. Anisotropic embedding spaces (the deep-dive above) can put the random-pair floor at cosine ≈ 0.7 — a retrieved-and-irrelevant document can score higher than a genuinely relevant one scores in a calibrated space. Compare any retrieval score against that embedder's own random-pair distribution before trusting an absolute threshold.

Run it yourself

Try it code/dot_product_geometry.py reproduces the king/queen/bicycle numbers above exactly, then samples random unit vectors at $d\in\{2,64,4096\}$ and prints the mean and std of 5,000 pairwise cosines — the same measurement Panel 2 makes live, extended to Qwen3's actual width.
Python
import torch

torch.manual_seed(42)

# king/queen/bicycle: the page's worked example, verified bit-for-bit
king    = torch.tensor([0.9, 0.8, 0.1])
queen   = torch.tensor([0.85, -0.7, 0.15])
bicycle = torch.tensor([-0.4, 0.05, 0.9])
cos = torch.nn.CosineSimilarity(dim=0)
print(king @ queen, cos(king, queen))   # 0.2200, 0.1639
