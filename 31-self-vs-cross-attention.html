Attention III: Self vs Cross — The Diffusion Hinge

Pages 29–30 built the soft lookup and its $\sqrt{d_{\text{head}}}$ thermostat, all with $K$ and $V$ coming from the same sequence the queries do. Change only that one thing — where $K,V$ come from — and self-attention becomes cross-attention. That single edit is the whole mechanism behind typing "a red car" into ComfyUI and getting a red car. You have watched this work a thousand times and never been shown the matmul that does it. Here it is.

🧵 Thread [THREAD: the dot product — beat 6]. Same atom as page 27's similarity meter, page 29's $QK^\top$ table, page 30's per-head directions. What is new: the score matrix is no longer forced square. When the keys come from a different sequence, $QK^\top$ is a table of dot products between two different sets of vectors — image patches down the rows, words across the columns. Reopen later: the diffusion track (pages 53–62) drives exactly this rectangular matrix when it conditions on your prompt.
🧵 Thread [THREAD: the memory budget — seed]. The reality-check thread from p.18. What is new: two numbers land here that you have felt physically — the 77-token CLIP limit you have hit when a prompt got long, and FLUX.2 at 32B, whose unoptimized footprint your unified memory swallows and a 24 GB card cannot. Reopen later: page 36 does the measured-budget arithmetic on your actual box.
🔮 Predict first When you type "a red car" and the car comes out red, the prompt reaches the image through attention. Is the image attending to the text, the text attending to the image, or are they concatenated and attending jointly? Commit to one before you scroll — you have the deepest hook on this page, because you have run the thing.

Hold that guess. The honest answer is "all three are right, for different model families" — and the reason that is not a cop-out is the whole point of this page. The classic (SD1.5/SDXL) answer is the first one: the image attends to the text. The 2026 (FLUX) answer is the third: concatenate and let self-attention sort it out. We build the first, then show the third is the same mechanism with the arguments glued together.

The only difference is where $K$ and $V$ come from

Self-attention, from page 29: every token makes its own query, key, and value from its own vector $\mathbf{x}_i$, and each token attends over the same set of tokens. Now hold the machinery fixed and ask a different question — let the queries come from one sequence and the keys/values from another. Nothing else changes. That is cross-attention.

Write it once, in frozen row-batched notation ($W$ is $(d_{\text{out}}, d_{\text{in}})$, all attention is row-batched — notation §1.3):

$$\text{CrossAttn}(X, Y) = \text{softmax}\!\left(\frac{(XW_Q^\top)\,(YW_K^\top)^\top}{\sqrt{d_{\text{head}}}}\right)\,(YW_V^\top).$$

Set $Y = X$ (so $S_{\text{txt}} = S$, $d_c = d$) and you are back to self-attention exactly — self is the special case of cross where the two sequences are the same one. Watch the shapes flow, and notice what stays fixed no matter how long the text is:

X (S, d) # query sequence — image patches Y (S_txt, d_c) # key/value sequence — text tokens (S_txt ≠ S, d_c ≠ d both fine) W_Q (d_head, d) # (out, in) — ALWAYS W_K, W_V (d_head, d_c) # these two absorb Y's width d_c Q = X @ W_Qᵀ (S, d_head) K = Y @ W_Kᵀ (S_txt, d_head) V = Y @ W_Vᵀ (S_txt, d_head) scores = Q @ Kᵀ / sqrt(d_head) (S, S_txt) # RECTANGULAR — one row per patch, one col per word A = softmax_row(scores) (S, S_txt) # each patch's row sums to 1 over the words out = A @ V (S, d_head) # S rows — one per query, ALWAYS (page 29 fact #1)
🔑 Key idea In a text-to-image diffusion model, the image is the query and the text is the key/value. Every image patch asks "which words are about me?"; the word embeddings answer; the patch pulls a blend of their values. When you type "a red car" and the car comes out red, this matmul is why — and it resolves the prediction: the image attends to the text, not the reverse.

Translation note: papers write the query position as $t$, $i$, $n$, or $j$; the sequence length $S$ as $T$, $L$, $N$, or seq_len; the text length as $S_{\text{txt}}$, context_len, or encoder_seq_len; and $d_{\text{head}}$ as $d_k$, $d_v$, or head_dim. Same object each time — only the letter changes.

✍️ Worked example — one patch reads three words Take one image-patch query $\mathbf{q}=(1.0,\,0.5)$ (so $d_{\text{head}}=2$, $\sqrt{d_{\text{head}}}=1.4142$) and three text keys $\mathbf{k}_{\text{red}}=(1.2,\,0.1)$, $\mathbf{k}_{\text{car}}=(0.3,\,0.9)$, $\mathbf{k}_{\text{sky}}=(-0.8,\,0.2)$. Raw dot products: $\mathbf{q}\!\cdot\!\mathbf{k}_{\text{red}}=1.25$, $\mathbf{q}\!\cdot\!\mathbf{k}_{\text{car}}=0.75$, $\mathbf{q}\!\cdot\!\mathbf{k}_{\text{sky}}=-0.70$. Scale by $\sqrt2$: $[0.8839,\,0.5303,\,-0.4950]$. Softmax → exponentials $[2.4203,\,1.6995,\,0.6096]$, sum $4.7294$, so the patch's attention row is $$A = [\,\mathbf{0.5118},\ \mathbf{0.3593},\ \mathbf{0.1289}\,]\quad(\text{sums to }1.0000).$$ This patch pulls 51.2% of its output from "red", 35.9% from "car", 12.9% from "sky" — a real, differentiable blend, the same softmax(QKᵀ/√d_head)V the demo below runs live per patch (there over its own wider $d_{\text{head}}=16$ and its own seeded keys, so its numbers differ — the operation is identical).

The shapes in a system you have actually run

A representative SDXL-class conditioning path (verify the exact widths against your diffusers build — they vary by model, so read these as the shape of the shapes, not frozen constants):

  • $Y$ = CLIP text embeddings, $(S_{\text{txt}}{=}\mathbf{77},\ 2048)$. That 77 is the CLIP context length every ComfyUI user has bumped into; the $2048$ is a concatenation of two CLIP encoders (CLIP-L $768$ + CLIP-G $1280$).
  • $X$ = flattened U-Net feature map, e.g. $(1024,\ 640)$ — $1024$ spatial positions of width $640$.
  • cross-attention score matrix $A$: $(1024,\ 77)$ — 1024 image positions × 77 text tokens, and each entry $A_{ij}$ is literally "how much does image-patch $i$ care about word $j$."

That $(1024, 77)$ matrix is the object your cross-attention-map, prompt-editing, and regional-prompting nodes reach in and rewrite. You have used those nodes; now you know they are writing into $A$.

→ Diffusion · this is the hinge the whole track swings on

Where $K,V$ enter decides everything downstream

Once you see that the condition enters only through $K$ and $V$, a pile of things you have used stop being magic and become consequences:

  • Classifier-free guidance (CFG). Its target is this $Y$-path: run the denoiser twice, once with $Y=$ your prompt and once with $Y=\varnothing$ (the empty condition), then extrapolate away from the unconditioned direction. CFG only means anything once you know $Y$ enters through $K,V$. (Sized and derived in the diffusion track, pages 53–62.)
  • IP-Adapter / ControlNet. Add a second cross-attention over image (or map) embeddings — "condition on anything you can embed" is just "give the patches a second set of keys to query."
  • The 77-token wall. A direct consequence of the old design: CLIP's context is 77, so $S_{\text{txt}}$ caps at 77 and long prompts get truncated or chunked. Not a law of attention — a property of that one encoder.
🔑 Key idea — the 2026 twist (resolves the prediction's third option) FLUX.1 and FLUX.2 use no separate cross-attention. Their MMDiT / joint attention concatenates the text tokens and the image tokens into one sequence and runs plain self-attention over it — so image↔image, image↔text, text↔image, and text↔text all live in one score matrix. Cross-attention is a special case of self-attention over a concatenated sequence. Teach cross first; reveal MMDiT as "just glue the two sequences together and let self-attention sort it out."

The concrete 2026 numbers, because the memory thread lands here:

  • FLUX.1-dev: ~12B params, MMDiT, dual text encoders (T5-XXL 4.7B + CLIP-L) [VP].
  • FLUX.2-dev: 32B params, a single Mistral-3 text encoder (context $\le 512$ tokens — so the 77-wall is gone), a 32-channel VAE at spatial factor $f=8$ [VP]. At bf16 the transformer weights alone are $32\times10^9\times 2 = \mathbf{64\ GB}$ [DER]; with the text encoder and working state the unoptimized pipeline runs well past 80 GB.

That is the motivating collision: FLUX.2 at >80 GB does not fit on a 24 GB card, but slots into your box's 128 GB unified memory with room to work. (The measured usable figure — it is not the full 128 — is page 36's story; don't quote 128 as usable yet.) The 77-token limit you have cursed at is a fossil of the old cross-attention design; FLUX removing it is a fossil of the new joint-attention one.

→ LLM · the other way to route K,V is by time-order — that is page 32

There are exactly two interesting things you can do to "where attention looks," and you have now seen one. Cross-attention routes $K,V$ from a different sequence. The other story routes attention within one sequence by time-order: position $i$ may see only positions $\le i$. That is causal masking, the hinge for the LLM track, and it gets its own page next (page 32) — the additive $-\infty$ mask, the "4096 training signals from one forward pass" economics, and the KV-cache payoff all live there.

⚠️ The fork contrast — draw this seam explicitly A diffusion U-Net's image self-attention is bidirectional / unmasked. Causality is a property of time-ordered generation — text is generated left to right, so an LLM masks the future — but an image is not time-ordered, so there is nothing to mask. The same attn() primitive is masked for an LLM and unmasked for a U-Net. The mask, not the mechanism, is what differs between the two tracks. Both inherit the identical function you built on page 29.

Four traps

⚠️ "Cross-attention is a different mechanism." Same function, different arguments. Self is attn(Q=x, K=x, V=x); cross is attn(Q=x, K=y, V=y). The code is identical — only the $K,V$ arguments (and therefore the column count of $A$) change. The demo and the code artifact show both call sites side by side.
⚠️ "The prompt is injected / mixed into the image." Backwards. The image queries the prompt — it is a pull, content-addressed by each patch, not a push. This is precisely why prompts are unreliable: you don't control the query. The patch decides what to ask for; the prompt only decides what is available to be asked.
⚠️ "$K,V$ must match $Q$'s width." No. $K$ must match $Q$ in $d_{\text{head}}$ (they are dotted together); $V$ must match $K$ in length $S_{\text{txt}}$ (one value per key). $Y$'s native width $d_c$ (2048 for CLIP, 4096 for T5, whatever Mistral uses) is free — $W_K, W_V$ project it down to $d_{\text{head}}$. That is exactly why you can bolt a T5, a Mistral, or a CLIP onto the same U-Net.
⚠️ "Diffusion U-Nets use causal masks like LLMs do." No — image self-attention is bidirectional / unmasked (the fork contrast above). Causality is about time-ordered generation, and images aren't time-ordered. Causal masking itself is dissected on page 32.

The code that proves the mechanism is identical

Try it — code/self_and_cross.py Calls one function — F.scaled_dot_product_attention — two ways: self (Q=K=V=x) and cross (Q=image, K=V=text of a different length). Prints both shape ribbons so you see the code is the same and only the $K,V$ arguments (and the column count of $A$) change.
Python
import torch
import torch.nn.functional as F

B, H, d_head = 1, 8, 64
S_img, S_txt = 1024, 77          # image positions vs CLIP text tokens

img = torch.randn(B, H, S_img, d_head)   # already projected + reshaped to heads
txt = torch.randn(B, H, S_txt, d_head)

# SELF: keys and values are the image itself
o_self = F.scaled_dot_product_attention(img, img, img)   # Q=K=V=img
assert o_self.shape == (B, H, S_img, d_head)              # (1, 8, 1024, 64)

# CROSS: same function — keys and values are the TEXT (a different length)
o_cross = F.scaled_dot_product_attention(img, txt, txt)  # Q=img, K=V=txt
assert o_cross.shape == (B, H, S_img, d_head)             # (1, 8, 1024, 64) — SAME rows

# the score matrix is where the difference shows: (S_img, S_img) vs (S_img, S_txt)
scale = d_head ** -0.5
A_self  = (img @ img.transpose(-2, -1) * scale).softmax(-1)   # (1,8,1024,1024) square
A_cross = (img @ txt.transpose(-2, -1) * scale).softmax(-1)   # (1,8,1024,77) rectangular
assert torch.allclose(A_cross.sum(-1), torch.ones_like(A_cross.sum(-1)))  # rows sum to 1
