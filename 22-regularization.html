The two curves — your primary diagnostic

Every training run you'll ever launch produces two numbers per checkpoint: the loss on the data you trained on, and the loss on data you held out. Plotted against each other, that pair is the single most informative readout in the whole loop — more informative than the train loss alone, which by itself cannot distinguish "learning" from "memorizing."

📐 Split by the right unit Before any curve means anything, the split has to be honest: dedupe (no row appears in both sets), no leakage across the boundary, and for instruction data specifically, split by source document, not by row — two rows drawn from the same document leak information about each other (shared phrasing, shared facts) even if the rows themselves are never duplicated. A val curve computed over a leaky split isn't measuring generalization; it's measuring how well the model memorized the training set, twice.

Plot both curves on the same log-y axis, and draw the random-guess baseline as a horizontal reference line — for a $V$-way classification problem the baseline is $\ln V$ nats (page 12); for the binary moons classifier in the demo below, $V=2$ and the line sits at $\ln 2 = 0.6931$ nats. Always plot the baseline. A curve that never gets meaningfully below it isn't learning — it's the underfitting shape, and no amount of staring at the gap between train and val will tell you that; only the baseline will.

Six shapes, six diagnoses

The two-curve plot has exactly six shapes worth recognizing. Each one names a different problem, and — this is the point of memorizing the table — each shape has exactly one fix. Reading the wrong shape as "overfitting" and reaching for weight decay when the real problem is a leaky split wastes a training run and teaches you nothing.

#ShapeDiagnosisFix
1Both curves high, flatUnderfitting — LR too low or model too smallRaise capacity or LR (page 13, page 19)
2Both curves low, small gapHealthyStop when val flattens
3Train → 0, val risesOverfittingRegularize / stop earlier / fewer epochs
4Train ≫ val (val lower than train)Leakage, or a val set too small/too easyFix the split
5Both plateau high after a few stepsA wiring bug or dead LR — not a generalization problemPage 24's overfit-one-batch test
6Val jagged, no trendVal set too small to estimateEnlarge it, or smooth over a window (page 16)
⚠️ "A rising val curve always means overfitting." Only if train is also low. Shape 4 (val below train) is leakage, not generalization; shape 5 (both flat and high from the start) is a bug, not a capacity problem. Read the shape, not just the direction of one line.

The regularizers — and which ones you actually need

Three standard tools sit under shape 3's fix. Here's what each one costs and buys, sized to a few-hundred to few-thousand-example fine-tune rather than a from-scratch pretrain:

📐 Weight decay, dropout, early stopping

Weight decay $\lambda$ — pulls every weight toward 0 each step; page 17 already gave you the numbers and the mechanism (decoupled, i.e. AdamW): $\lambda=0.1$ for LLM fine-tuning, $\lambda=0.01$ for diffusion, applied to weights only — never to biases, norm gains, or (usually) embeddings. Not re-derived here; see page 17's ledger.

Dropout — randomly zeroes activations during training so no single unit can be relied on. Standard in 2010s vision nets; largely off in 2026 LLM fine-tuning (still seen in some vision / LoRA-dropout setups). It solves a problem — co-adapted units in a huge, from-scratch, many-epoch training run — that a small, low-rank, few-epoch fine-tune mostly doesn't have.

Early stopping — evaluate every ~50 steps, keep the checkpoint with the best val loss, stop when it stops improving. Costs nothing extra to compute (you were already running val) and directly targets shape 3's fix: it stops before the rise, rather than trying to prevent the rise from happening at all.

Resolving the PREDICT: for a few-hundred-example fine-tune, early stopping + fewer epochs + a low rank/LR is the whole toolkit. Weight decay helps a little at the margin; dropout is usually irrelevant in 2026 LLM fine-tuning; and "more epochs" is the wrong direction entirely — you're already past the peak of shape 3, and every additional epoch pushes val higher. Page 23 explains why this is the right toolkit for this regime, via the bias–variance decomposition; this page just gives you the toolkit.

Catastrophic forgetting — the failure the two-curve plot won't show you

⚠️ "Val loss looks fine, so fine-tuning is safe." The task's train/val curves can look perfect — low loss, small gap, textbook shape 2 — while the model gets measurably worse at everything it wasn't fine-tuned on. This is catastrophic forgetting, and it does not appear on the task's own two-curve plot, because that plot only ever measures the task. The only way to see it is to also track loss on a held-out set of general data (not the fine-tuning task) and watch that curve separately. Mitigations: mix in some general data, lower the rank/LR, fewer epochs — the same knobs that reduce overfitting, deployed against a problem the overfitting plot can't detect. (Developed fully on page 24, whose instrumentation panel logs both curves side by side.)

Three more traps

⚠️ "Dropout is essential regularization." In 2026 LLM fine-tuning it's usually off. Early stopping and a low rank do more work for less cost, and dropout's noise can actively hurt a short, low-data fine-tune where every gradient signal is precious.
⚠️ "More epochs will fix a rising val curve." It makes shape 3 worse, not better — every epoch past the val minimum is pure memorization of the training set. If the curve is rising, the fix is to stop earlier, not train longer.
⚠️ "A healthy task curve means the fine-tune is safe to ship." Not if you haven't checked catastrophic forgetting (above). A perfect shape-2 curve on the fine-tuning task is necessary, not sufficient.
