Three things people confuse autograd with — and what it actually is

📐 Autograd is none of the first two

(a) It is not symbolic differentiation. It never produces a formula for the gradient. There is no d/dx algebra engine inside PyTorch rewriting $\tanh$ into $1-\tanh^2$ as a symbol.

(b) It is not numerical (finite) differences. It never perturbs a weight by a tiny $\epsilon$ and re-runs the forward pass. No $\epsilon$, no re-run, no approximation error from step size.

(c) It is the chain rule, executed on a recorded graph, in reverse. Every operation you run forward silently records its inputs and a rule for its local derivative. .backward() walks that record backward, multiplying and accumulating local slopes — exactly the four equations you applied by hand on page 14, ordered by a machine instead of by you. [THREAD:CHAIN]

The recording is called the tape (or the computation graph). Each elementary op — a multiply, an add, a $\tanh$ — stores two things when it runs forward: the input values it consumed, and a small closure _backward that knows how to convert an incoming gradient into gradients for its inputs. Calling loss.backward() visits the ops in reverse topological order, seeding the output with $\partial\mathcal{L}/\partial\mathcal{L}=1$ and pushing gradients backward until every leaf (every parameter) has its .grad filled. nn.js's Value engine — the one running every demo in this course — does literally this; you can open assets/nn.js and read the _backward closures.

🔬 Deep dive: reverse mode returns all $P$ gradients in one pass; forward mode would cost $P$ passes

A chain of derivatives can be walked two ways. Forward mode pushes $\partial\cdot/\partial\theta_j$ forward from one input $\theta_j$: one pass gives how every quantity depends on that one input. Reverse mode pulls $\partial\mathcal{L}/\partial\cdot$ backward from the one scalar output: one pass gives how the loss depends on every input at once.

Training has exactly one scalar loss and millions-to-billions of parameters. So reverse mode is the only affordable choice: one backward pass yields all $P$ gradients, where forward mode would need $P$ separate passes. That asymmetry — one output, many inputs — is the entire reason .backward() exists and runs the graph the direction it does. (Page 14 proved the same cost story for the pencil: one backward pass, not $P$ forward perturbations.)

The nine pencil numbers, reproduced by .backward()

Build TN-1 as a two-layer network in PyTorch, load the frozen weights, run one forward pass to the loss, and call loss.backward(). Out come nine .grad tensors. Eight of them match page 14's pencil values to every printed decimal. The ninth is the story of this page.

📐 The mandated cross-checks — and why the tolerances differ

Input 1 ($x=[1.0,2.0]$, the dead-unit case, constants.md §8.5):

Python
assert torch.allclose(model[0].weight.grad,
    torch.tensor([[-0.3764, -0.7527], [0.2028, 0.4056]]), atol=1e-4)

Note $-0.7527$, not $-0.7528$. With the brief's old $-0.7528$ the error is $9.66\times10^{-5}$ against a budget of $1.075\times10^{-4}$ — a 10% margin that survives only because atol=1e-4 happens to be loose enough to swallow a rounding slip. Fixing the last digit drops the error to $3.4\times10^{-6}$, a 30× margin — a nine-page thread that no longer leans on a coincidence.

Input 2 ($x=[0.60,-0.20]$, the healthy case, constants.md §8.7) — carried to 5 decimals at atol=1e-5, not 1e-4:

Python
assert torch.allclose(model[0].weight.grad,
    torch.tensor([[-0.13475, 0.04492], [0.22140, -0.07380]]), atol=1e-5)

Why the tighter tolerance? Input 2's smallest gradient is $\partial\mathcal{L}/\partial W_1[0][1]=+0.0449$. At 4 decimals, a check on a number that small has a margin of only $\sim\!2\times$ — the same trap as Input 1. Five decimals at 1e-5 restores real headroom. The precision you assert at must match the magnitude you are asserting about.

★ The float that isn't zero

Eight gradients matched. The ninth — Input 1's $\partial\mathcal{L}/\partial W_2[0]$, the dead-unit entry you proved was exactly $0$ — comes back from float32 autograd as 1.4020191230201817e-08. This is the answer to the PREDICT, and it is not a bug:

🔑 Both are right By hand you got exactly 0. Torch prints 1.4e-08. Both are right. 0.5 − 0.6 + 0.1 is not zero in binary floating point — it is the textbook non-associativity demo, and you just hit it for free. This is the lesson of the logsumexp page come back around: the math on paper and the math in the machine are not the same math. The gradient is zero to seven decimals, the unit is dead, and the lesson stands — but the number on your screen will not be 0.0, and a course that told you it would be is a course that has never run its own code.

Trace it to the source. The dead unit was dead because its pre-activation was $z_{1,1}=0.5(1.0)+(-0.3)(2.0)+0.1$ — algebraically $0.5-0.6+0.1=0$. But in binary floating point, $0.3$, $0.6$, and $0.1$ are each already rounded (none is representable exactly), and the three-term sum is evaluated left-to-right, so a rounding residue survives:

Python · constants.md §8.4
z_11 = 0.5*1.0 + (-0.3)*2.0 + 0.1        # the classic 0.5 - 0.6 + 0.1
  float64  →   2.7755575615628914e-17       # not 0.0
  float32  →  -2.2351742e-08              # not 0.0

torch float32  dW2 → [1.4020191230201817e-08, -0.5021151900291443]

The dead-unit entry is 1.4020191230201817e-08 — the "exactly zero" gradient, alive at the eighth decimal because $a_{1,1}=\tanh(z_{1,1})$ was fed a $z_{1,1}$ that was itself $\sim\!10^{-8}$, not $0$. Propagate that through .backward() and the local products, accumulated in reverse-topological order, preserve the residue. The second entry, $-0.5021151900291443$, rounds to page 14's printed $-0.5021$ — it is the same number, seen at full float precision. The paper value is the idealisation; the float value is what actually ran. [THREAD:CHAIN]

That was the lesson stated as a precaution. This is the lesson made literal on a number you computed by hand: not a warning about a value you might have seen, but the value itself, off by 1.4e-08, on your own box.

Why zero_grad() exists — and what happens when you forget it

PyTorch does not overwrite .grad on each backward pass — it accumulates into it. That is a deliberate feature: it lets you split one large effective batch across several backward passes and sum their gradients before stepping, which is how you train a batch that does not fit in memory in one shot (page 18's gradient accumulation). [THREAD:MEM] — the same mechanism, seen here as a footgun, is reopened there as a memory trick.

The price of that feature: if you forget to call zero_grad() between steps, step $k$ does not use the gradient of batch $k$ — it uses the running sum of every gradient so far. Over $k$ steps of roughly similar magnitude the used gradient grows like

$$\tilde{\mathbf g}_k \;=\; \sum_{j=1}^{k} \mathbf g_j \;\approx\; k\,\mathbf g,$$

so the effective learning rate climbs with every step, and the update turns into a runaway. Instead of descending, the loss thrashes — an oversized, stale gradient yanks the weights across the landscape, so the curve dips and then spikes by orders of magnitude, over and over. It is not training; it is being thrown around.

⚠️ "You will see the gradient come out as exactly 0.0." False in every float precision. The dead-unit gradient is $0$ on paper and 1.4e-08 in float32; both are correct, and the second is what your machine reports. A related trap: "a nonzero float means the computation is wrong." It is correct to seven digits — the paper $0$ is the idealisation, the float is the measurement.
⚠️ "Forgetting zero_grad() has no effect — I saw a run converge without it." That is the full-batch illusion. On full batches every accumulated gradient points the same way, so the growing sum just acts like a rising learning rate along a consistent descent direction — it can even converge faster. The bug only reveals itself with mini-batches, where each accumulated gradient is a stale sum over different batches pointing in different directions. The demo below is mini-batch for exactly this reason.
