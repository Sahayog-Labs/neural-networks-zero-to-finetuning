Trunk callback — nothing new here

The trunk already derived everything structural. The adapter replaces the weight update with a rank-$r$ factorization:

$$h = W_0\,x \;+\; \frac{\alpha}{r}\,B\,A\,x, \qquad A \in \mathbb{R}^{r \times d_{in}},\quad B \in \mathbb{R}^{d_{out}\times r},\quad r \ll d_{in},d_{out}$$

SymbolIsShapeFrom
$W_0$the frozen pretrained weight — never touched, still fully resident$(d_{out}, d_{in})$the base checkpoint
$A$LoRA "down" projection, init $\mathcal{N}(0,\varsigma^2)$$(r, d_{in})$trained
$B$LoRA "up" projection, init zeros — so $BA = 0$ at step 0$(d_{out}, r)$trained
$r$rank — a capacity knob, a count not an indexscalaryou choose it
$\alpha$LoRA scaling (lora_alpha); $\alpha/r$ is the effective scalescalaryou choose it

Because $B$ starts at zeros, the adapter begins as an exact no-op — $h = W_0 x$, bit-identical to the base model — and only earns its influence through training. The $\alpha/r$ scale, the $B{=}0$ init, and the claim that the update $\Delta W$ (not $W_0$) is what's low-rank were all established on the trunk's LoRA-math and SVD/rank pages. [THREAD:TN-1 beat: LoRA target] — this is the same $\frac{\alpha}{r}BA$ update you attached to TN-1's $W^{(2)}$ by hand, now attached to Qwen3's linear matrices instead of a 9-parameter toy.

🔑 The predict, resolved Just the matrices. The equation is character-for-character the same one that drives your ComfyUI LoRAs. What differs for an LLM is entirely which matrices you attach $A$ and $B$ to, which rank you pick, and a short list of LLM-specific gotchas. That is the whole content of this page — everything else you already have.

All-linear, and the number that forces it

A Qwen3-8B transformer block has seven linear matrices worth adapting: four in attention (q_proj, k_proj, v_proj, o_proj) and three in the feed-forward network (gate_proj, up_proj, down_proj). Count their parameters:

per block
attention (q,k,v,o)        41,943,040   # constants §1.2 [DER]
MLP (gate,up,down)        150,994,944   # constants §1.2 [DER]  ← 78.26% of the block
targetable per block      192,937,984   # the 7 linears (norms are not LoRA targets)

The three MLP matrices are 150,994,944 params = 78.26% of the block [DER, constants §1.3]. (State the denominator: that's 78.26% of a block; measured against the whole model the FFN is 66.4% — both are true, and quoting the wrong one starts arguments.) Either way, if you target attention only, you have frozen roughly three-quarters of every block and then wondered why the fine-tune didn't take.

📐 Default: target_modules="all-linear" Attach the adapter to all seven linear projections in every block [VP]. It is the peft default for a reason: the MLP is where most of the trainable surface lives, and the 78.26% number is why "just do q/v" is stale advice.
⚠️ "Attention is where the knowledge is — target q_proj/v_proj only" The single most persistent piece of stale LoRA advice, inherited from the original 2021 paper's ablations on far smaller models. You just counted it: the MLP is 78.26% of the block. Targeting attention-only freezes that 78% and leaves the highest-capacity matrices untrained. Default to all-linear; the demo below lets you watch the trainable count collapse when you switch back to attn-only.

So LoRA $r=16$ all-linear trains 43,646,976 params = 0.533% of P [DER, constants §3]. The adapter file, at bf16, is $43{,}646{,}976 \times 2 = \textbf{87.3 MB}$ — smaller than a single one of your ComfyUI LoRAs, for a model 10× the size.

✍️ The base count closes to the byte (constants §3, D-07) The 6,945,767,424 in the LoRA table above omits the norms (they aren't adapter targets, correctly), so it is not the model's non-embedding count. Add the pieces back and it reconciles exactly — no rounding: $$6{,}945{,}767{,}424 \;+\; \underbrace{36\times8{,}448}_{304{,}128\ \text{norms}} \;+\; \underbrace{4{,}096}_{\text{final norm}} \;+\; 2\times622{,}329{,}856 \;=\; \mathbf{8{,}190{,}735{,}360}$$ No page in this course says "closes to three significant figures." It closes to the byte, and that it does is the point — this is the arithmetic you can trust your hardware against.

The 187× — an optimizer-state trick, not a model-compression trick

Here is the sentence to memorize, verbatim:

LoRA is not a model-compression technique, it's an optimizer-state-compression technique. The base model didn't shrink — all 16.38 GB is still resident. The trainable state shrank by 187×, which is exactly $8{,}190{,}735{,}360 \div 43{,}646{,}976$, because optimizer state is linear in trainable parameters. That's the whole trick.

Trainable state alone goes 114.67 GB → 0.61 GB = 187× [DER, constants §2.3], and 187 is the parameter ratio ($8{,}190{,}735{,}360 / 43{,}646{,}976 = 187.7$). Counting the frozen base weights that still sit in memory, the full-footprint comparison is 131.05 GB → 17.08 GB = 7.67× (state-to-state, both excluding activations). [THREAD:memory ledger] — the wall you hit at 122.05 GiB on p.49 becomes 15.91 GiB, comfortably inside your measured 121.6875 GiB box.

⚠️ "LoRA cuts my activation memory too" It does not. LoRA freezes weights; it does not shorten the graph. You still run a full forward and backprop through the entire network, so activation memory at a given batch and sequence length is essentially unchanged between full FT and LoRA (constants §2.4). Every LoRA-vs-full comparison in this course is state-to-state, with activations named as a separate, identical line on both sides. When you measure your real footprint it will sit above 17.08 GB by exactly the activation term — and now you knew that first.

Rank follows the dataset, not the model

🔑 The rule Rank should scale with the information content of your dataset — not the size of your model. A bigger base model does not want a bigger $r$.

For LLM SFT that lands at $r = 64$–$256$. Push it to the extreme: $r=256$ all-linear is 698,351,616 params = 8.53% of P → about 27.6 GB total [DER, constants §3] — which still trivially fits your box. What goes wrong at high rank on an LLM? Essentially nothing; rank is nearly free. What goes wrong at low rank? It underperforms a full fine-tune because it hasn't the capacity. So stop being stingy with rank.

(The diffusion track sits at the opposite end of the same rule: 15–20 training images, $r=8$–$32$, and high rank memorizes. Same principle — rank tracks dataset information — opposite regime. Your ComfyUI intuition is the low-data end of this line.)

⚠️ "α is a strength dial I tune separately" $r$ is capacity; $\alpha$ is scale, and the effective scale is only ever the ratio $\alpha/r$ — it is not a second capacity knob. $\alpha$ and the learning rate are nearly redundant: raising $\alpha$ and raising LR do almost the same thing. Hold $\alpha = 2r$ (or a fixed $\alpha=16$) and tune the learning rate instead. Speaking of which: LLM LoRA wants LR ≈ 1e-4 – 3e-4 [EST, constants §9.4], roughly 10× a full fine-tune's — tolerable precisely because $B{=}0$ at init and the update is scaled down by $\alpha/r$.

QLoRA — the 70B on your desk

Quantize the frozen base to 4-bit NF4 and you shrink the one thing LoRA left at full size: the resident base weights. NF4 with double-quant costs 0.515869 B/param = 4.127 bits/param [VP, constants §3]. For Qwen3-8B the NF4 base is $8{,}190{,}735{,}360 \times 0.515869 = \textbf{4.225 GB (3.935 GiB)}$ [DER], and QLoRA total lands around 4.9–7 GB → 19–26× vs a full fine-tune.

⚠️ Double-quant is OFF by default bnb_4bit_use_double_quant defaults to False in both bitsandbytes and BitsAndBytesConfig [VP, constants §3]. Un-opted-in NF4 is 4.5 bits = 0.5625 B/param → 4.607 GB for Qwen3-8B, not 4.225 GB. Every QLoRA memory figure here assumes you explicitly set it =True. Set it.
🔑 The payoff of the whole track Llama-3.3-70B QLoRA, $r=16$ all-linear = 207,093,760 params = 0.294% → 3.31 GB state → ≈ 39.7 GB total with an NF4 base [DER, constants §3]. A 70-billion-parameter model, fine-tuning, on a 240 W box on your desk. That specific combination — 128 GB unified memory plus QLoRA — is what puts a 70B in a hobbyist's hands. On the Spark, though, an 8B should use plain LoRA (~17 GB); QLoRA trades compute for memory (it dequantizes every forward pass and is slower), so reserve it for the 70B.

The 2026 PEFT menagerie

All shipping in peft 0.19.1 [VP]. Names to recognize, not lore to master:

  • DoRA (use_dora=True) — splits the update into magnitude + direction; helps most at low rank, so it buys little once you're at $r=256$.
  • rsLoRA (use_rslora=True) — rescales by $\alpha/\sqrt r$ instead of $\alpha/r$; starts to matter at $r \ge 64$.
  • PiSSA / EVA / CorDA / LoftQ (init_lora_weights=...) — smarter initializations of $A,B$ than "random + zeros"; a warm start, not a new algorithm.
  • target_parameters for MoE — Hugging Face experts are raw nn.Parameters, so all-linear silently skips them; name them explicitly (D-06 caveat).

The run

Measured 8B LoRA throughput on the Spark is 6,969.59 tok/s [VP, constants §6.7]. Your actual project — 2,000 examples × 800 tokens × 3 epochs = 4.8M tokens — is then $4{,}800{,}000 / 6{,}969.59 = \textbf{11.5 minutes}$, using about 17 GB of your box. (A 70B QLoRA runs at 759.79 tok/s → the same tokens take ~1.75 hours.)

🔑 The bottleneck was never compute Eleven and a half minutes. The constraint on this fine-tune was never your GPU — it was the dataset you built on p.48, and the prior question of whether you should be fine-tuning at all (p.47). The machine was ready the whole time.
