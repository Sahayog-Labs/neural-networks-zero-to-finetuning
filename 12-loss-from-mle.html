What “Learning” Optimizes: Loss from Maximum Likelihood

A loss is not a design choice pulled from a menu. It is the negative log-likelihood of your data under the model — one principle, and cross-entropy and mean-squared error are its two branches. This is the page where “the surface we descend” gets its shape.

🔮 Predict first A freshly-initialised Qwen3-8B — every weight random, no training yet — is asked to predict the next token from a vocabulary of 151,936. On the very first batch, what loss (in nats) do you expect? Write a single number down before you scroll. (You are allowed to be wrong by a lot. The point is to have a number to be wrong against.)

Hold that number. By the end of the first section you will be able to compute the answer in one line, and it will turn out to be a quantity you can read off the vocabulary size alone. [THREAD:QWEN — the anchor model, beat 1]

Learning means making the data probable

Strip away the machinery and a trained model is a claim about probability: it assigns a number $p_\theta(y\mid x)$ to every possible answer $y$ for an input $x$. “Learning” is nudging the parameters $\theta$ until the answers that actually happened in your data get high probability. That is the whole game, and it hands us a loss for free.

A good loss is a scalar that is small when the model is confident and right, and large when it is confident and wrong. Notice the asymmetry already: being wrong is not what costs you — being wrong while certain is. We will see that fall directly out of the logarithm.

🔑 Key idea There is one derivation and two branches. Pick a probability model for your output, write down the probability it assigns to the observed target, take the negative logarithm, and average. Everything the course ever calls a “loss” is this quantity with a different probability model plugged in.

Maximum likelihood, in one line

Assume the $N$ examples are independent. The probability the model assigns to the entire dataset is the product of the per-example probabilities. We want the parameters that make that product as large as possible:

$$\theta^\star=\arg\max_\theta\;\prod_{n=1}^{N} p_\theta(y_n\mid x_n) \;\;\Longleftrightarrow\;\; \theta^\star=\arg\min_\theta\;\underbrace{-\sum_{n=1}^{N}\ln p_\theta(y_n\mid x_n)}_{\text{the loss}}$$

Two moves turn the left side into the right. Taking a logarithm turns the product into a sum (and does not move the location of the maximum, because $\ln$ is monotone). Negating turns “maximise likelihood” into “minimise a loss,” because everyone’s optimizer walks downhill. The negative log-likelihood is the loss. We did not invent it; we recognised it.

Units matter and the course fixes them once: because we used the natural logarithm, loss is measured in nats. Use $\log_2$ and you would get bits. Same quantity, different ruler; this course always uses nats.

Branch C — a categorical output gives cross-entropy

Suppose the model outputs a probability over $V$ discrete classes via a softmax of its logits $z$ (this is exactly the softmax you built on page 9). The probability of the true class $c$ is $\hat y_c$, so the per-example loss is:

$$\mathcal{L}=-\ln \hat y_c=-\ln\frac{e^{z_c}}{\sum_{v=1}^{V} e^{z_v}} = \Big(\ln\textstyle\sum_v e^{z_v}\Big) - z_c$$

The rightmost form — $\big(\ln\sum_v e^{z_v}\big)-z_c$ — is worth a second look. It never divides, never forms a probability. That is not an accident; it is the shape the machine actually computes, and we come back to why at the bottom of the page.

✍️ Worked example — the course’s canonical logits Take $z=[\,2.0,\ 1.0,\ 0.1\,]$ with true class $c=0$ (the first). Every number below is frozen — you will meet this exact triple again on the temperature, top-$k$, and $\sqrt{d_{\text{head}}}$ pages.
Exponentiate: $e^{z}=[\,7.389056,\ 2.718282,\ 1.105171\,]$, whose sum is $\ \sum_v e^{z_v}=11.212509$.
Normalise: $\hat y=[\,\mathbf{0.659001},\ 0.242433,\ 0.098566\,]$ — these are probabilities, they sum to $1$.
The loss is the negative log of the true class’s probability: $$\mathcal{L}=-\ln(0.659001)=\mathbf{0.417030}\ \text{nats.}$$
And the gradient the next chapter lives on: $$\frac{\partial\mathcal{L}}{\partial z}=\hat y-y=[\,\mathbf{-0.340999},\ +0.242433,\ +0.098566\,].$$ Look at those three numbers: they sum to exactly zero. They always will — softmax outputs sum to $1$, the one-hot target sums to $1$, so their difference sums to $0$. Probability that leaves the true class is exactly the probability that arrives at the others.
⚠️ “The softmax gradient must be complicated” It looks like it should be — a softmax couples every logit to every other through the shared denominator. But the whole Jacobian collapses: for softmax followed by cross-entropy, $\partial\mathcal{L}/\partial z=\hat y-y$, full stop. It is arguably the cleanest gradient in all of deep learning, and it is why the output layer is never the hard part of backprop (page 14).

Branch A — a Gaussian output gives mean-squared error

Now change only the probability model. Suppose the target is a real number and the model predicts it with Gaussian noise around its guess: $y\sim\mathcal{N}(\hat y,\sigma^2)$ — this is the exact bell curve you met on page 8, now with the model’s prediction $\hat y$ sitting in the mean slot. Write its negative log:

$$-\ln p(y\mid\hat y)=-\ln\!\left[\frac{1}{\sqrt{2\pi}\,\sigma} \exp\!\Big(-\frac{(y-\hat y)^2}{2\sigma^2}\Big)\right] =\frac{1}{2\sigma^2}(y-\hat y)^2+\underbrace{\ln\big(\sqrt{2\pi}\,\sigma\big)}_{\text{constant in }\hat y}$$

$y$the observed real-valued targetscalarthe dataset

$\sigma$noise standard deviation — a fixed scale, not a sigmoidscalar > 0page 8’s Gaussian
$-\ln p$the per-example loss for this modelscalar, natsdefined here

The term with $\hat y$ in it is $\tfrac{1}{2\sigma^2}(y-\hat y)^2$; the rest is a constant that the gradient never sees. So minimising the negative log-likelihood of a Gaussian output is minimising $\tfrac12(y-\hat y)^2$ — squared error, up to the fixed scale $1/\sigma^2$.

🔑 Key idea — MSE is not a separate fact Mean-squared error is Branch A of the same derivation: it is literally the negative log of the page-8 Gaussian. Cross-entropy is Branch C, for a categorical output. They differ in one place only — the assumed noise distribution — and in nothing else. The bell curve you met on page 8 is where squared error comes from. (When the diffusion track arrives, its training loss will be a third corollary of this same Gaussian — noise prediction is Branch A wearing a different hat.)
⚠️ “Cross-entropy and MSE are unrelated losses” They feel unrelated because they are taught in different weeks and look nothing alike. But both are negative log-likelihoods; they differ only in the assumed output distribution — categorical (softmax) for cross-entropy, Gaussian for MSE. Choose the distribution, and the loss is no longer yours to pick.

The anchors: numbers a loss curve is measured against

A loss value in isolation means nothing. To read a training curve you need reference lines — the loss of a model that has learned nothing. For a $V$-way softmax that has not been trained, every class is equally likely, so $\hat y_c=1/V$ and the loss is $-\ln(1/V)=\ln V$.

📐 Loss anchors (memorise these three)
  • Random-init loss, Qwen3-8B: $\ln V=\ln(151{,}936)=\mathbf{11.93}$ nats. This is the number your PREDICT was chasing. [THREAD:QWEN — beat 2]
  • Perplexity at init: $\mathrm{PPL}=e^{\mathcal{L}}=e^{\ln V}=V=\mathbf{151{,}936}$ — exactly the vocabulary size, by construction.
  • Coin flip ($V=2$): $\ln 2=\mathbf{0.6931}$ nats.
Always plot $\ln V$ as a horizontal line on every loss curve. A curve sitting on that line has learned nothing; a curve that never leaves it means the run is broken. It is the single cheapest sanity check in the course.
⚠️ Not 11.76 You will see $\ln V=11.76$ quoted in blog posts and even some course notes. That is $\ln(128{,}256)$ — Llama-3’s vocabulary, not Qwen3’s. This course’s anchor model is Qwen3-8B with $V=151{,}936$, so the number is 11.93. If your untrained run starts near 11.76 you are looking at the wrong tokenizer.
⚠️ “Higher accuracy always means lower loss” Loss is calibrated confidence, not correctness. A model that predicts the right class but only at $\hat y_c=0.34$ still carries loss $-\ln 0.34=1.08$ nats even though it is “right.” You can nudge accuracy up while loss goes up, and vice versa. Track the loss; it is what the gradient actually sees.

Two things to feel in your hands. Drag $z_0$ down and the loss climbs without bound — being confidently wrong is unboundedly expensive — yet the gradient $\partial\mathcal{L}/\partial z$ never leaves $[-1,1]$, because it is just $\hat y - y$ and probabilities live in $[0,1]$. Why the LogSumExp trick exists

On paper, cross-entropy forms $\hat y_c=e^{z_c}/\sum_v e^{z_v}$ and takes a log. In the machine, $e^{z}$ overflows: a logit of $1000$ — ordinary during training — makes $e^{1000}=\infty$ in float, and the loss becomes nan before it becomes a number. The fix is an identity, not an approximation:

$$\ln\sum_{v} e^{z_v}=m+\ln\sum_{v} e^{z_v-m},\qquad m=\max_v z_v.$$

Subtracting the max makes the largest exponent exactly $e^0=1$, so nothing overflows, and because we add $m$ back the value is identical — the constant carries no error and no gradient. That is why the fused form $\mathcal{L}=\big(\ln\sum_v e^{z_v}\big)-z_c$ from the Branch-C box never divides: this course’s crossEntropyLoss (and PyTorch’s) computes the log-sum-exp directly and never forms the probability at full scale.

🔑 The same lesson, arriving early LogSumExp is the first appearance of a theme this course returns to made-literal on page 15: the math on paper and the math in the machine are not the same math. On paper, $e^{1000}$ is a large finite number; in float it is $\infty$. The stable algorithm is the one that respects the difference. (Page 15 shows the flip side — a gradient that is exactly $0$ on paper prints as 1.4e-08 — and it is the same lesson.)
⚠️ “LogSumExp makes cross-entropy differentiable” No — cross-entropy was already differentiable; its gradient is the tidy $\hat y-y$ we derived above. LogSumExp changes nothing about the value or the derivative. It only changes whether the computation overflows. It is a numerical-stability trick, not a mathematical one.
🔬 Deep dive: why $\ln\sum_v e^{z_v}$ is called a “soft” maximum

As one logit $z_c$ pulls far above the rest, $\ln\sum_v e^{z_v}\to z_c=\max_v z_v$: the sum is dominated by its largest term. So $\ln\sum_v e^{z_v}$ is a smooth, differentiable stand-in for $\max_v z_v$ — hence “LogSumExp is a soft max,” and the reason the function that normalises it is called softmax. The $m=\max_v z_v$ we subtract for stability is the very quantity the whole expression approximates — a pleasing symmetry, and a reason the subtraction is exact rather than a hack.

Run it yourself

Try it — code/12_cross_entropy.py Reproduce the frozen triple in PyTorch. The one rule that matters: cross_entropy takes logits, never probabilities — feed it log(softmax(...)) and you double-count the log. The assertion nails the value to the sixth decimal.
Python
import torch
import torch.nn.functional as F

# The course's canonical logits (constants.md 9.2), true class 0.
z = torch.tensor([[2.0, 1.0, 0.1]])   # shape (1, 3) = (batch, classes)
c = torch.tensor([0])                # the TRUE class index

# F.cross_entropy expects LOGITS, not probabilities. It fuses
# log_softmax + nll_loss with the LogSumExp trick inside.
loss = F.cross_entropy(z, c)
print(loss.item())            # 0.41703... nats
assert abs(loss.item() - 0.417030) < 1e-5

# The gradient is exactly softmax(z) - onehot(c), and sums to 0.
z.requires_grad_(True)
F.cross_entropy(z, c).backward()
print(z.grad)              # [[-0.340999, 0.242433, 0.098566]]
assert abs(z.grad.sum().item()) < 1e-6

# The random-init anchor: an untrained V-way softmax scores ln(V).
import math
print(math.log(151936))     # 11.93  <- Qwen3-8B, NOT 11.76 (that's Llama-3)

The only thing that ever changes is what we ask it to predict. Cross-entropy over a vocabulary is an LLM’s loss; squared error on injected noise is a diffusion model’s. One principle, two branches, the whole course.
