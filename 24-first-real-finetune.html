Your First Real Fine-Tune — and the Twelve Ways Training Fails

Pages 12–23 handed you a loss, a backward pass, an optimizer, a schedule, and a regularization toolkit as separate topics. The training loop, grown by accretion

Every line below traces to a page you've already read. Nothing here is new machinery — it's the same nine objects, assembled. The three highlighted lines are the ones the last three pages (19, 21, 22) bolted on; everything dimmed is load-bearing but old news.

Python
model.train()
for epoch in range(N_EPOCHS):                              # p.16: epochs over mini-batches, not one big batch
    for xb, yb in train_loader:                          # p.16: mini-batch, sampled fresh each step
        opt.zero_grad()                                     # p.15: skip this and grads ACCUMULATE
        logits = model(xb)                                       # forward pass — TN-1's shape, p.6
        loss = loss_fn(logits, yb)                                # p.12: NLL — cross-entropy or MSE
        loss.backward()                                      # pp.14-15: reverse-mode autograd fills .grad
        torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)  # p.19: cap rare spikes, AFTER backward, BEFORE step
        opt.step()                                            # p.17: SGD → AdamW, the actual parameter update
        scheduler.step()                                       # p.21: η_k is a function of the step, not a constant
    model.eval()
    val_loss = evaluate(model, val_loader)                       # p.22: watch BOTH curves, not just train
    if val_loss < best_val:                                     # p.22: keep the BEST checkpoint, not the last one
        best_val, best_state = val_loss, model.state_dict()
    model.train()

Resolving the predict: the 60-second test is overfit one batch of 8 examples to ~0 loss, dropout and weight decay off, before you touch N_EPOCHS, η, or anything else in the loop above. If it collapses toward zero, every line you just read is wired correctly and any remaining problem is about data or generalization. If it doesn't, you have a bug — and no amount of learning-rate tuning will fix a bug. This bisects the entire space of training failures in one run, which is why it comes before anything else on this page.

🔑 The universal first move Overfit 8 examples, dropout / augmentation / weight decay off, ~200 steps. Loss → 0: wiring is correct, your problem is data or generalization, not the loop. Loss ↛ 0: a bug — checking zero_grad(), the loss function, and the label alignment costs less than one wasted training run.
⚠️ Three things people believe about training that aren't true
  • "Training failures are mysterious." There are about twelve, each with a distinct signature — the three tables below are the field guide.
  • "If the loss is going down, it's working." Catastrophic forgetting and data leakage both hide behind a perfectly healthy task-loss curve (see §14.3 below).
  • "Tune the LR first." Overfit one batch first — LR tuning cannot fix a wiring bug, and tuning on top of a bug just wastes runs while looking productive.

The instrumentation panel — log every run, always

📐 Log these eight things, every training run (training §14.4)
  1. train + val loss, with the $\ln V$ baseline drawn on the same axes
  2. gradient global norm — grad norm leads loss. A spike shows up here one or two steps before it shows up in the loss curve.
  3. the actual LR read back from the scheduler (not the config value) — catches the "scheduled-by-epoch-not-step" bug (p.21)
  4. update ratio $\|\Delta\theta\|/\|\theta\| \approx 10^{-3}$ ([EST], a rule of thumb — Karpathy's CS231n notes, never formally validated, constants.md §9.3)
  5. dead-unit fraction
  6. VRAM high-water mark (torch.cuda.max_memory_allocated()) — predict from page 18's ledger, then measure [THREAD:MEM]
  7. throughput (steps/s)
  8. wall-clock per step

✍️ Worked example — reading the update ratio Suppose at some step the flattened trainable-parameter vector has norm $\|\theta_k\| = 812.4$, and that step's update moved it by $\|\Delta\theta_k\| = 0.81$. The update ratio is $$\frac{\|\Delta\theta_k\|}{\|\theta_k\|} = \frac{0.81}{812.4} = \mathbf{0.000997} \approx 1.0\times10^{-3}.$$ Squarely on the healthy rule of thumb. If that ratio read $10^{-1}$ instead, the LR is roughly 100× too high; if it read $10^{-6}$, roughly 1000× too low (constants.md §9.3, [EST]).
🔑 The determinism trick (training §14.5) A failure at the same step, every run, with a fixed seed is a data problem — some specific example is poison. A failure at a different step each run is numerics or a race — an overflow that depends on which random draw happened to line up badly. One bisection, and you know which half of the codebase to read. Saves hours.

The milestone deliverable — a real fine-tune on your box

The browser demo proves the loop; it cannot prove the machine, because a toy MLP with 100-odd parameters doesn't touch a GPU. The script below does — it fine-tunes Qwen3-0.6B, the anchor model's smoke-test sibling (constants.md §1.4), with a real peft LoRA adapter, and the arithmetic transfers to the Qwen3-8B anchor by swapping four config numbers: model name, nothing about $r$ or target_modules changes. It runs the overfit-one-batch check first, prints the trainable-param count, logs the instrumentation panel, and predicts peak VRAM from the page-18 ledger formula before measuring it with max_memory_allocated() — write your own guess down before you run it; the gap between your guess and the measured number (activations, allocator overhead) is the point, not a failure of the formula (constants.md §2.2 makes the same disclaimer about the frozen 122.05 GiB figure).

⚠️ The fp32-default trap (constants.md §7.3, [VP]) SFTTrainer still defaults to fp32 when model is passed as a string, even though from_pretrained would have inferred bf16 from the checkpoint. Two libraries, one script, opposite defaults. The script below always passes model_init_kwargs={"dtype": torch.bfloat16} explicitly — do the same in anything you write yourself.
