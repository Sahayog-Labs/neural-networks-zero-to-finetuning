Part IV — Adaptation

Number Formats & QLoRA

A pretrained weight matrix is sixteen million numbers clustered near zero. You do not need sixteen bits to store one of them — you need about four, provided you place the levels where the mass is and re-anchor the scale often enough that one outlier can't ruin the neighbourhood. That re-anchoring, not the bit-width, is what modern quantization is.

🔮 Predict, before you read on You quantize a weight matrix to 4 bits. What cuts the error most: (a) using a cleverer 4-bit number format, (b) re-anchoring the scale every 16 weights instead of every 4096, or (c) they're about equal? Commit — then watch the quantizer demo settle it.

The one-sentence frame

Everything on this page unpacks a single claim: 4-bit names a budget, not a scheme, and almost all of the quality lives in the scheme — specifically in how finely you re-scale. QLoRA is the payoff. It is the trick that took the LoRA you derived on page 40 and shrank the frozen base underneath it from 16.38 GB to about 4 GB, which is the difference between a 70B living on your desk and a 70B living in someone else's data centre.

A float is sign, range, and precision

A floating-point number is three fields: a sign bit, an exponent that sets the range (how big and how small it can be), and a mantissa that sets the precision (how many significant bits inside that range). Trade bits between the last two and you get every format in the table below.

📐 Range vs precision — the formats you will actually meet

The one thing to remember: bf16 has fp32's exponent range and far fewer mantissa bits; fp16 has a much smaller range and its gradients underflow to zero below ~6e-8, which is why fp16 training needs a loss-scaling GradScaler. bf16 keeps the range, so it needs none. Use bf16 on the Spark, always. fp16's max of 65,504 and fp32's machine epsilon of 1.19e-7 are both [VP, constants §9.3].

Why did the GradScaler era end? Because bf16 gave you fp32's range in half the bytes; the scaler existed only to drag fp16 gradients up out of the underflow floor before they rounded to zero. On Blackwell you keep the range for free, so the scaler is a museum piece — mention it once, in the past tense, and move on.

Quantization is a scale times a code

Every low-bit weight format — INT4, NF4, FP4 — is the same two-part object. You store a small integer code per weight, and one scale shared across a block of weights. The reconstructed weight is their product:

📐 The general form of a quantized weight $$w \;\approx\; s \cdot q, \qquad q \in \{\text{one of } 2^{\,b_q}\text{ code levels}\},\quad s = \text{one scale per block};$$ $$\text{effective bits/param} \;=\; b_q \;+\; \frac{b_s}{\text{block size}}.$$
SymbolIsShape / unitsFrom
$w$the real weight you are compressingscalar (bf16 originally)the pretrained checkpoint
$q$the code — which of the $2^{b_q}$ levels this weight snapped toa $b_q$-bit integerstored per weight
$s$the block scale — the stretch that maps codes back to weight unitsscalarstored once per block, $=\max|w|/\max|q|$
$b_q$bits per code (4 for NF4/FP4, 8 for INT8/FP8)integerthe format
$b_s$bits per stored scale (32 for fp32, 8 for fp8 or a power-of-two exponent)integerthe format

Read the second line: the code costs $b_q$ bits per weight, but the scale's $b_s$ bits are amortized over the whole block. Shrink the block and you buy accuracy — at the cost of storing more scales. That trade is the entire game.

QLoRA's NF4 budget, to the bit

QLoRA freezes the base model and stores it in a 4-bit format called NF4 (NormalFloat-4), then trains an ordinary LoRA adapter (page 40) on top in bf16. The frozen base is read-only, so it can be as compressed as you can bear. Here is exactly how many bits per parameter NF4-with-double-quant costs — the derivation matters because the number that half the internet copies is an arithmetic mistake.

✍️ Effective bits per parameter, NF4 + double-quant

Three costs, added:

  • The code: NF4 weight $= 4$ bits.
  • First-level scale: one fp8 absmax scale per 64-weight block $\Rightarrow 8/64 = 0.125$ bits/param.
  • Second-level scale (the "double" in double-quant): those fp8 scales are themselves grouped and re-scaled — one fp32 constant per 256 scales, and each scale covers 64 params, so the fp32 constant amortizes over $64\times256 = 16{,}384$ params $\Rightarrow 32/16{,}384 = 0.00195$ bits/param.

Overhead $= \dfrac{8}{64} + \dfrac{32}{64\cdot 256} = \dfrac{65}{512} = \mathbf{0.127}$ bits. Total $= 4 + \dfrac{65}{512} = \dfrac{2113}{512} = \mathbf{4.127}\ \text{bits} = \mathbf{0.516}\ \text{B/param}$ [VP, constants §3 · arXiv 2305.14314 §3].

Cross-check against the paper's own figures: the saving from double-quant is $0.5 - 0.127 = 0.373$ bits/param; on a 65B model that is $65\text{e}9 \times 0.373 / 8 = \mathbf{3.03\ GB}$, matching the paper's abstract ("approximately 3 GB"). Two routes, one number.

⚠️ The 4.25-bit / 0.53-B trap You will see "$4 + 8/64 + 32/256 = 4.25$ bits $\Rightarrow 0.53$ B/param" quoted everywhere. It is an arithmetic error, not a rival convention. It divides the second-level fp32 constant by 256 — the number of scales it covers — instead of by $64\times256 = 16{,}384$, the number of parameters those scales cover. That single dropped factor of the 64-block inflates $0.00195$ into $0.125$. There is no range and no second convention: the paper's own arithmetic gives 4.127 bits, and so does the bitsandbytes source.
✍️ QLoRA on Qwen3-8B — the footprint

With double-quant on, the frozen NF4 base is

$$8{,}190{,}735{,}360 \times 0.515869\ \text{B/param} = \mathbf{4.225\ GB\ (3.935\ GiB)}\quad\text{[DER]}.$$

Why NF4 and not plain INT4

A pretrained weight tensor is approximately $\mathcal{N}(0,\sigma^2)$ — a bell curve centred on zero. Plain INT4 spreads its 16 levels evenly across the range, so most of them land out in the near-empty tails while the dense middle gets only a few. NF4 instead places its 16 levels at the quantiles of a standard normal: every code is equally likely to be used, so the levels crowd together near zero where the weights actually are. Same 4 bits, the codes just moved to where the mass is. [THREAD:the dot product — beat: the shape of the data] — the weights' distribution, not the bit-count, is what decides the format.

Resolving the PREDICT, one paragraph in: it's (b), by a wide margin. Swapping the 4-bit format (INT4 → NF4) trims the error a little; dragging the block size from 4096 down to 16 collapses it, because a single heavy-tail outlier no longer inflates the scale for thousands of its neighbours. Then drag the bit-width (INT4 → INT8) and watch it matter less than the block size did. Granularity beats precision. That is the one quantization lesson worth keeping, and it is why the whole field re-scales blockwise.

🔑 The NVFP4 story — granularity, not mantissa MXFP4 forces every block's scale to a power of two (the E8M0 exponent field). A block that wants a scale of 3 rounds to 2 or 4 and eats up to 33% error before a single weight is quantized. NVFP4 allows an arbitrary FP8 scale and halves the block to 16. That is the whole difference between them — and it is worth about 88% of the error. The general lesson, one more time: quantization error is dominated by scale granularity, not by mantissa bits. The Spark has NVFP4 in hardware, and bitsandbytes ships a native sm_121 wheel for its aarch64 + CUDA-13 stack [VP, constants §6.9] — so the 70B QLoRA path below actually runs. Here the Spark is better served than an x86 box.

Three things "4-bit" does not mean

⚠️ "4-bit means 16 evenly-spaced values" No modern format does this. NF4 uses normal quantiles, E2M1 is a tiny float with unevenly spaced representable values, GGUF mixes bit-widths across a tensor. Even spacing is exactly the INT4 the demo showed you losing to NF4.
⚠️ "Quantization is faster because there's less math" For decode it is the same math — you dequantize and then multiply. The win is reading 4× fewer bytes, and single-token decode is memory-bound, so bytes are what bind. Quantization is a bandwidth optimization wearing a compute costume. (Caveat: NVFP4 tensor cores execute FP4 matmuls natively, so prefill and training — which are compute-bound — do get a real compute win.) [THREAD:the dot product — beat: memory-bound] — decode reads every weight once and does ~2 FLOP per weight, which is precisely the roofline story page 43 measures.
⚠️ "Train in the format you'll serve in" No. Train NF4/bf16, then merge the adapter into the original bf16 weights and re-quantize for serving (GGUF/AWQ/NVFP4). Merging a bf16 adapter into an already-NF4 base is lossy — the base has already thrown away the precision the merge needs.

QLoRA buys memory with compute

QLoRA is roughly 20–40% slower than plain LoRA, because you pay a dequantization on every forward pass. You use it when LoRA does not fit, not for speed. On the Spark, an 8B LoRA already fits comfortably at 17.08 GB, so 8B QLoRA is the wrong choice — it would trade speed for memory you do not need. QLoRA earns its keep at 70B:

✍️ A 70B on your desk

Llama-3.3-70B, NF4 base with double-quant $\approx 36.4$ GB, plus an $r=16$ all-linear adapter of 207,093,760 params (0.294% of the model) $\approx 3.31$ GB of state:

$$36.4\ \text{GB} + 3.31\ \text{GB} \approx \mathbf{39.7\ GB\ total}\quad\text{[DER, constants \S3]}.$$

Convert to compare against the box: $39.7\ \text{GB} \times 0.9313 \approx 37.0$ GiB, comfortably inside the measured 121.6875 GiB [MEA-DEV, constants §6.8]. 128 GB of unified memory plus QLoRA is the specific combination that puts a 70B on your desk. [THREAD:Qwen3-8B — the anchor]: the 0.294% and 207,093,760 come from the same $r(d_{in}+d_{out})$ arithmetic you ran for the 8B on page 40, just with the 70B's dimensions.

🔬 Paged optimizers, and why the Spark is the exception

bitsandbytes offers paged optimizers: on a memory spike, optimizer state is paged out to CPU RAM to avoid an out-of-memory crash, the way an OS pages to disk. Every tutorial you read will recommend them for QLoRA on a discrete GPU, because there the GPU's VRAM and the host's RAM are two separate pools connected by a slow PCIe bus.

On the Spark there is no separate host pool to page to. Its 128 GB is unified — the CPU and GPU address the same physical memory coherently over NVLink-C2C. A "page to CPU" is a no-op move within one pool. This is one of many places a discrete-GPU tutorial will quietly mislead you about your own hardware; the fix is to know the machine, which is the whole posture of this course.
