Three ways to change a model's behaviour — and where this one sits

You have already met two of the three post-training verbs. On page 48 SFT taught imitation — copy these answers — and DPO taught preference — this answer beats that one. The third is the subject of this page:

SFT teaches imitation. DPO teaches preference. RLVR teaches reasoning wherever a program can check the answer.

RLVR — reinforcement learning from verifiable rewards — is the most actionable of the three, because "a program that checks the answer" is a thing your own domain almost certainly has. Does the config parse? Does the SQL run? Does the extracted JSON match the schema? Does the unit test pass? Each of those is a ground-truth reward with no human in the loop and no learned reward model to drift or be gamed. If you have a checker, you can do real reinforcement learning.

🔑 Key idea This is where the 2026 reasoning models come from. Not from a bigger preference dataset — from a reward you can compute. Where the answer is checkable, you no longer need a human to say which of two completions is better; the checker says so, for free, a thousand times a second.

What DeepSeek actually used: GRPO's group-relative advantage

Reinforcement learning needs one hard number per response: how much better than average was this? Classic PPO answers it with a second neural network — a value network (a "critic") trained alongside the policy just to predict the average reward from a state. It is half the parameters, half the tuning headaches, and half the failure modes.

GRPO (Group Relative Policy Optimization) throws the critic away. For a single prompt it samples a group of $G$ responses, scores each with the verifier, and defines each response's advantage relative to the group's own mean and spread:

$$A_i \;=\; \frac{r_i - \operatorname{mean}(r_1, \dots, r_G)}{\operatorname{std}(r_1, \dots, r_G)}$$

📐 The insight — ship it verbatim PPO needs a learned value network to know "was this better than average?" GRPO just asks $G$ times and takes the mean. The critic was replaceable by a sample mean — and that is why GRPO took over.

Notation note: here $i$ indexes the $G$ responses in the group, not a sequence position — this is one of the few places the course reuses $i$ as a group index, matching how the GRPO papers write $A_i$. The rewards $r_i$ come from a program, not a model.

Why this is still the chain rule [THREAD:chain rule beat: policy gradient]

Same verb, one last time. On page 4 it was a derivative; on page 14 it was backprop through TN-1; on page 48 it was the SFT/DPO gradient. The policy-gradient update is the same chain rule applied to a new loss — push up the log-probability of responses that scored above the group mean, push down the ones below it:

$$\nabla_\theta \mathcal{L} \;=\; -\,\mathbb{E}\!\left[\, A_i \, \nabla_\theta \log \pi_\theta(o_i \mid x) \,\right]$$

SymbolIsShapeFrom
$\nabla_\theta \mathcal{L}$the gradient we descend$(P,)$ — same shape as $\theta$the optimizer step, as always

$\pi_\theta(o_i \mid x)$the policy's probability of response $o_i$ given prompt $x$scalar in $(0,1)$the model being trained (page 48's $\pi_\theta$)
$\nabla_\theta \log \pi_\theta$the score of the log-prob — the chain rule, computed by autograd$(P,)$backprop (page 15)

No new machinery. The advantage $A_i$ is just a scalar multiplier on the same $\log\pi_\theta$ gradient you have differentiated since page 14. The clever part is entirely in where the multiplier comes from — a group average instead of a trained critic.

✍️ One row, by hand Suppose a group's rewards average to $\operatorname{mean}=0.4$ with $\operatorname{std}=0.5$. A response that passed the checker has $r_i = 1$, so its advantage is $A_i = (1 - 0.4)/0.5 = 0.6/0.5 = \mathbf{1.2}$ — it scored 1.2 standard deviations above the group, and the policy gets nudged toward it by exactly that much. A response that failed ($r_i = 0$) gets $A_i = (0 - 0.4)/0.5 = -0.8$ — nudged away. (This is the number the quiz asks for.)

Why this capstone needs a container, not a pip install

SFT and DPO are offline: the data sits in a file, you stream it through. GRPO is online — it must generate the group of $G$ responses inside the training loop, score them, then step. Generation-in-the-loop is the expensive part, and it is exactly what gated this capstone until vLLM was unblocked on the Spark's stack.

The validated path is the vllm/vllm-openai:cu130-nightly container, pinned by digest [constants §6.9]. The reason is sharp and worth internalizing:

⚠️ The CUDA-12 wheel trap The PyPI aarch64 vLLM wheel is built against CUDA 12, and DGX OS is CUDA-13-only. Any package pinning libcudart.so.12 fails at import — before a single kernel's compute-capability is even checked. This is the real aarch64 trap on this box: not "kernels don't exist," but "the wrong CUDA runtime shared library isn't there." The container ships its own CUDA-13 userspace and sidesteps it entirely. [constants §6.9, VP]

Two dependencies that make the quantized path viable here, both verified on the box: bitsandbytes ships a native sm_121 build for ARM64 + CUDA 13 (x86 does not get it — the Spark is, for once, the better-served platform), and TRL v1 flipped one relevant default:

📐 The one TRL v1 break to know GRPOConfig's vllm_mode default changed "server" → "colocate" [constants §7.2]. Colocate runs generation and training in the same process — right for a single box. Bump the pin (trl>=1.8,<2) and move on; the rest of the v0→v1 migration is, in TRL's own words, minimal.

Runnable scope: a 0.6B tonight, the 8B by substitution

Because you generate inside the loop, the capstone runs on Qwen3-0.6B — the smoke-test sibling [constants §1.4] — so a real generation-in-the-loop job finishes in minutes rather than hours. The arithmetic is identical to the 8B anchor; you recover the 8B's memory and throughput numbers by swapping four config values — $d_{\text{model}}$, $L$, $H_{kv}$, $d_{\text{ff}}$ — into the same formulas from page 49. For the anchor those are $4096,\ 36,\ 8,\ 12288$ [VP, constants §1.1]; the 0.6B is smaller on every axis, which is precisely why it fits a two-hour experiment.

Point the reward at something your own domain can check — a reasoning_accuracy_reward, a JSON-schema validator, a math grader — and the model gets measurably better at a task you can verify, with no human labels and no reward model.

⚠️ "GRPO is magic / RLVR installs knowledge" It does neither. Like all post-training (pages 47 and 48), GRPO reweights behaviours the base model already has toward the ones the verifier scores highly. It cannot install a fact the base never learned — attempting that is the same failure as fine-tuning on documents to "teach facts" (page 47), and it produces the same confident-but-wrong output. RLVR works only where a program can score the output. Where it can, it is remarkable; where it can't, it has nothing to optimize.
