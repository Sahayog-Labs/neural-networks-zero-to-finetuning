Part III · Architecture

Architecture Is Inductive Bias

Pages 1–24 taught you one architecture: the plain MLP, dense layer after dense layer. It trained (page 17), it fit in a memory ledger (page 18), it fine-tuned for real (page 24). So why does every model you actually use — the U-Net under your ComfyUI workflow, the transformer under every LLM — look nothing like it? Not because an MLP can't represent what they represent. Because it can't learn it from the data and compute you actually have. This page makes that distinction precise, then prices it.

🔮 Predict first A 224×224 RGB image goes into a fully-connected layer with 1,000 hidden units. A conv layer with 64 filters of size 3×3×3 processes the same image. Which has more parameters, and by roughly what factor — 10×, 1,000×, or 100,000×? Commit to one before you scroll.

A hard-coded prejudice about your data

An architecture is a hard-coded prejudice about your data — a shortcut if you're right and a wall if you're wrong. Here's the sentence that matters for you specifically: an MLP is a universal approximator, so architecture is never about what a network can represent. It's entirely about what it can learn from finite data in finite time. Every architectural choice below is a bet about your data, installed before training starts.

📐 The universal approximation theorem, stated then punctured For a wide-enough hidden layer, some choice of weights makes a network match any continuous function to arbitrary precision: $$\Big|\sum_{i=1}^{N} w_i\,\phi\!\left(\mathbf{a}_i^\top\mathbf{x}+b_i\right)-f(\mathbf{x})\Big|<\varepsilon$$ That's an existence claim — it says weights exist, and says nothing about how large $N$ has to be, whether SGD ever finds those weights, or how much data it takes to find them. Architecture is entirely about those three questions. Representability was never the issue.

The killer worked example

Same image, two layers, count the parameters each one owns.

✍️ Worked example — the 84,000× gap

Dense layer 1, on the flattened image. Input dimension $224\times224\times3=150{,}528$. A dense layer to 1,000 hidden units owns one weight per (input, output) pair: $$150{,}528\times1{,}000=150{,}528{,}000\approx1.5\times10^{8}\ \text{params}\ \ (\approx301\text{ MB in bf16})$$

Conv layer, 64 filters of $3\times3\times3$. A filter owns one weight per (input channel, kernel position) pair, shared across every spatial location, plus one bias per filter: $$64\times(3\times3\times3)+64 = 64\times27+64 = 1{,}728+64=\mathbf{1{,}792}\ \text{params}$$

$$\frac{150{,}528{,}000}{1{,}792} = \mathbf{84{,}000\times}$$ Exactly — $1{,}792\times84{,}000=150{,}528{,}000$. 1,792 vs 150,528,000 — an 84,000× gap, in one layer. On the PREDICT above, the honest closest bucket is 100,000× (the real ratio, 84,000×, sits between the 1,000× and 100,000× options but an order of magnitude nearer the top one on a log scale).

The lesson isn't "conv is smaller." It's what the smallness buys: the FC layer could rediscover one edge detector at all 50,176 pixel positions independently, from data, if you gave it enough examples and enough time. The conv layer is handed translation-equivariance — the same 27-weight stencil is reused everywhere by construction, not learned into place.

The transformer row is the thesis of this page: its installed bias is unusually weak — "look everywhere, decide what matters from the data itself" is barely a prior at all compared to a conv kernel's hard-coded locality. That weakness is exactly why it needs so much data and compute to get going, and exactly why, once it has them, it stops being capped by a prior that doesn't fit the task. Page 38 (Part IV) makes this quantitative with real scaling-law numbers.

Three ways this goes wrong in your head

⚠️ "A bigger MLP can do anything a CNN can, so CNNs are just an optimization." True about representation (the UAT above), false about learning. Fix by arithmetic, not argument: 1,792 vs 150,528,000, one layer, same image. The FC layer isn't wrong, it's just handed a search problem 84,000× larger than it needs to be.
⚠️ "More inductive bias is better." Bias is a bet, not a virtue — it caps you exactly where it stops matching the data. Vision Transformers beat CNNs on image classification only once pretraining data crosses roughly 100 million images ([EST] — a folklore threshold from the ViT paper's data-scaling curves, not a frozen constants.md figure); below that, the CNN's baked-in locality prior wins because there isn't enough data to learn locality from scratch. Which architecture wins flips with dataset size — that is the opposite of "more bias, always better."
⚠️ "Architecture is where the intelligence lives." In 2026, nearly every frontier open model shares one block (see the thread box above). If architecture were where the intelligence lived, they'd all perform identically — they don't, because what differs is scale, data, and training, not the substrate. Architecture is the substrate the intelligence is built on, not the intelligence itself. (Page 35 assembles the block; page 38 makes "scale, data, training" quantitative.)
