Logs — only the identities that earn their place

Four identities carry everything on this page. You already know them; the point is which four, and why the first one is the entire reason log-space exists in machine learning.

$$\log(ab)=\log a+\log b,\qquad \log(a^n)=n\log a,\qquad e^{\log x}=x,\qquad \log_b x=\frac{\ln x}{\ln b}.$$

The first turns a product of many probabilities into a sum of log-probabilities — and a sum does not underflow. That is the whole trick, and we'll feel why in a moment.

📐 Convention: $\log=\ln$, and loss is in nats In machine learning $\log$ means the natural log $\ln$, always — never base 10, never base 2. A loss measured with $\ln$ is in nats. Want bits instead? Divide by $\ln 2 = 0.6931$. And perplexity is just the loss exponentiated, $\mathrm{PPL}=e^{\mathcal L}$: a loss of $2.0$ nats is a perplexity of $e^{2.0}\approx7.39$ — "as confused as picking uniformly among $7.4$ words."

Translation (what papers call these). We index the vocabulary with $v \in [0, V)$ and write the class count $V$. Most ML texts write $K$ for the number of classes and vocab_size in code; in this course $k$ is reserved for the optimizer step, so classes get $v$/$V$. The fact "$\ln V$ nats at initialization" is the same fact those texts write "$\ln K$."

Underflow, felt before it's fixed

Here is the crash the PREDICT box asked about. Scoring a sequence means multiplying its per-token probabilities. Do it naively and the machine does not give you a small number — it gives you nothing.

✍️ The product that dies

Multiply $0.1$ by itself 500 times:

$$\prod_{i=1}^{500} 0.1 = 10^{-500}.$$

The smallest normal float32 is about $1.18\times10^{-38}$; the smallest subnormal is about $1.4\times10^{-45}$. So $10^{-500}$ is not "very small" — it is exactly 0.0 in float32 (and in bf16, which shares the same exponent range). Then log(0.0) = -inf, and -inf * 0 = nan, and your run is dead on step 1.

Now do it in log-space instead — turn the product into a sum with $\log(ab)=\log a+\log b$:

$$\sum_{i=1}^{500}\ln(0.1) = 500 \times (-2.302585) = \mathbf{-1151.29}.$$

A perfectly ordinary floating-point number. Nothing overflowed, nothing underflowed. The two lines compute the same quantity, $\ln\big(\prod 0.1\big)=\sum\ln 0.1$ — one is a corpse, the other is fine.

The log-sum-exp trick — exact, not a hack

Softmax needs $\sum_j e^{z_j}$, and exponentials overflow just as eagerly as products underflow: $e^{88.7}$ already exceeds the float32 maximum of about $3.4\times10^{38}$. The fix you will see in every framework's source is to subtract the largest logit before exponentiating:

$$\log\sum_{j} e^{z_j} \;=\; m + \log\sum_{j} e^{\,z_j - m},\qquad m=\max_j z_j.$$

SymbolIsShape / unitsFrom
$z_j$the $j$-th logit (any real number, dimensionless)scalarthe network's output
$m$the largest logit, $\max_j z_j$scalarcomputed from $z$
$e^{\,z_j-m}$shifted exponential — the biggest is $e^0=1$, the rest $\le 1$scalarafter subtracting $m$
$V$number of classes / vocab size (the sum runs $j=1..V$)scalar$151{,}936$ for Qwen3-8B

After subtracting $m$, the largest exponent is $e^0=1$ and every other term is at most $1$ — they can only underflow to $0$, which is harmless in a sum. Crucially, this identity is exact: it is algebra, not an approximation. State that out loud, because it looks like a hack and is not one.

⚠️ Why F.cross_entropy wants logits, not probabilities This is exactly why F.cross_entropy(logits, targets) takes raw logits and not a probability vector: it fuses softmax + log + NLL into one numerically-stable kernel that does the max-subtraction for you. Writing log(softmax(x)) as two separate ops is a real bug — it works on friendly inputs and returns nan on the large logits a mid-training network actually produces. Never hand a loss function probabilities when it will take logits.

Softmax — turning logits into a distribution

Softmax takes a vector of logits and squashes it into a probability distribution: everything positive, summing to one.

$$p_i=\operatorname{softmax}(z)_i=\frac{e^{z_i}}{\sum_{j=1}^{V} e^{z_j}}.$$

SymbolIsShape / unitsFrom
$z$logits — the raw network outputs (any real, dimensionless)$(V,)$, batched $(B,S,V)$the final linear layer
$p_i$probability assigned to class $i$; $p_i>0$, $\sum_i p_i=1$scalar in $(0,1)$softmax of $z$
$\sum_j e^{z_j}$the normalizer (partition function) — one stable sum via log-sum-expscalarthe denominator
$V$vocab / class countscalar$151{,}936$ (Qwen3-8B)

Two properties do real work later, and you'll turn both of them yourself in the demo:

  • Shift invariance: $\operatorname{softmax}(z+c)=\operatorname{softmax}(z)$ — adding the same constant to every logit changes nothing. Only differences between logits matter. This is the very same fact that licenses subtracting the max above.
  • Temperature: $\operatorname{softmax}(z/T)$. As $T\to0$ the biggest logit wins outright (greedy, one-hot); at $T=1$ you get the model's own distribution; as $T\to\infty$ everything flattens to uniform. You have turned this exact slider in ComfyUI and every LLM UI — [THREAD:paper-vs-machine] it is page 1's "magnitude = confidence" fact wearing a sampling hat.

The three probabilities sum to $1$ (check: $0.659001+0.242433+0.098566=1.000000$). The loss is simply the negative log of the single probability you handed the right class.

Cross-entropy is just "make the data likely"

The general cross-entropy between a target distribution $y$ and a prediction $p$ is $\mathcal L_{\text{CE}}=-\sum_{i=1}^{V} y_i\log p_i$. In classification the target is one-hot — a $1$ on the correct class, $0$ everywhere else — so every term but one is multiplied by zero and the whole sum collapses:

$$\mathcal L_{\text{CE}}=-\sum_{i=1}^{V} y_i\log p_i \;\xrightarrow{\text{one-hot }y}\; \mathcal L=-\log p_{\text{correct}}.$$

🔑 The formula, demystified For a one-hot target, cross-entropy is nothing but the negative log of the single probability you assigned to the right token. The other $V-1$ terms vanish. There is no cleverness to memorize — the intimidating sum is one number in disguise.

And it isn't a loss someone dreamed up. Start from the likelihood of your whole dataset, $\prod_n p_\theta(y_n\mid x_n)$ — you want to make it large. But a product of many probabilities underflows (you just watched it die). Take logs, flip the sign to make it a minimization, and you have exactly $-\sum_n\log p_\theta(y_n\mid x_n)$ = negative log-likelihood = cross-entropy. Cross-entropy isn't invented; it's "make the data likely," rewritten so a computer can survive it.

📐 Forward pointer, not a duplicate [THREAD:loss-unification beat 1] The full maximum-likelihood derivation — and the surprise that MSE is the same principle under a Gaussian output assumption (seeded on page 8) — is page 12's job. Here we plant the seed: an LLM's next-token loss and a diffusion model's MSE are both negative log-likelihood under different assumed output distributions. Two losses, one idea.

The miracle cancellation

Differentiate cross-entropy-after-softmax with respect to the logits and something remarkable happens. The softmax's messy derivative and the log's $1/x$ annihilate each other, and what survives is the simplest gradient in the course:

$$\frac{\partial\mathcal L}{\partial z_i}=p_i-y_i\qquad(\text{"predicted minus actual"}).$$

For the canonical logits with true class $0$:

$$\frac{\partial\mathcal L}{\partial z}=\hat y - y=[\,0.659001-1,\ 0.242433,\ 0.098566\,] =[\mathbf{-0.340999},\ +0.242433,\ +0.098566].$$

🔑 Why cross-entropy beat MSE for classification The gradient is predicted minus actual: push the right class's logit up, push the rest down, each in proportion to how wrong you were. It sums to zero, always — a direct consequence of shift invariance, and a free sanity check. And it does not vanish: a confidently-wrong output has $p_{\text{correct}}\to0$, so the gradient magnitude $\to 1$ — maximally wrong means maximally corrected. Pair MSE with a sigmoid instead and that same confidently-wrong output gets a tiny gradient and barely learns. That contrast is the whole reason classification uses cross-entropy.

We state $p-y$ here and use it; we don't yet prove it. It is earned twice more: page 12 derives it from maximum likelihood, and page 14 rederives it as the very first gradient of the backprop pass, where it is the network's "grace moment."

A number you can check on step 0

📐 Random-init loss $=\ln V$ A freshly-initialized model has no idea which token comes next, so it spreads its probability uniformly across all $V$ classes: $p_{\text{correct}}=1/V$, and $\mathcal L=-\ln(1/V)=\ln V$. For Qwen3-8B with $V=151{,}936$: $$\mathcal L_{\text{init}}=\ln(151{,}936)=\mathbf{11.93}\ \text{nats},\qquad \mathrm{PPL}=e^{11.93}=V\ \text{exactly}.$$ If step 0 of a training run doesn't print about $11.9$, something is already wrong — a genuinely useful first check. (It is not $11.76$; that is Llama-3's $V=128{,}256$, retired here, constants.md §9.1.) Four traps worth naming

⚠️ "Softmax outputs are calibrated confidences" They sum to $1$ by construction — that is arithmetic, not calibration. Modern networks are systematically overconfident, and whether large LLMs are well-calibrated in 2026 is genuinely contested (Guo et al. 2017 showed image classifiers are miscalibrated; RLHF appears to degrade calibration further, but it's unsettled). Treat a $0.92$ softmax output as a ranking, not a promise.
⚠️ "Temperature changes what the model knows" It changes only the sampling decision. The logits are identical at every temperature; $T$ reshapes the distribution you draw from, not the network. Verify it yourself: same prompt and seed, different $T$ — same logits, and the +3 button in Demo 2 makes the analogous point for shift.
⚠️ "Softmax is a soft max" It's a soft argmax — it returns a distribution over which index wins, not the winning value. The name is a historical misnomer; reading it as "soft argmax" prevents a surprising amount of confusion.
⚠️ "Cross-entropy is a classification thing, irrelevant to diffusion" An LLM's next-token prediction is a $V$-way classification, and cross-entropy is its only loss. Diffusion uses MSE — but MSE is itself the negative log-likelihood of a fixed-variance Gaussian, so both are the same "make the data likely" principle under different output assumptions. Page 12 collects the unification; page 8 planted it.
🔬 Deep dive: bf16 vs fp16 — why range beats precision for training

Both are 16-bit floats, but they split the bits differently. bf16 keeps $8$ exponent bits — the same dynamic range as float32, about $10^{\pm38}$ — and spends only $7$ bits on the mantissa. fp16 keeps $10$ mantissa bits (more precision) but only $5$ exponent bits, so its range is a cramped $\sim6\times10^{-5}$ to $65{,}504$.

Training gradients span many orders of magnitude, and the small ones matter. fp16's narrow range makes them underflow to zero, which is why fp16 training needs a GradScaler to shove values back into range — and why bf16 needs no such crutch. bf16 trades precision it can spare for range it cannot. On your Blackwell GB10, bf16 is the default for exactly this reason. (fp16's ceiling of $65{,}504$ is the same wall the log-sum-exp trick keeps softmax away from: constants.md §9.3.)

The code that survives a big logit

Try it · code/softmax_stable.py Run it on CPU (<1 s). It computes softmax two ways on the canonical logits (they agree — $p_0=0.659001$, $\mathcal L=0.417030$, gradient sums to $0$), then on $[800,1,1]$ where the naive version returns nan and the stable one is fine. The last line shows F.cross_entropy reproducing the loss straight from logits.
Python
import torch
import torch.nn.functional as F

def softmax_naive(z):
    e = torch.exp(z)              # exp(800) -> inf
    return e / e.sum()          # inf/inf -> nan

def softmax_stable(z):
    z = z - z.max()             # exact shift; biggest exponent is e^0 = 1
    e = torch.exp(z)
    return e / e.sum()

z = torch.tensor([2.0, 1.0, 0.1])
p = softmax_stable(z)
print(p)                        # [0.659001, 0.242433, 0.098566]
print((-torch.log(p[0])).item())  # 0.417030  (true class 0)

y = torch.tensor([1., 0., 0.])
print((p - y))                  # gradient p - y, sums to 0.0

big = torch.tensor([800., 1., 1.])
print(softmax_naive(big))     # tensor([nan, nan, nan])
print(softmax_stable(big))    # tensor([1., 0., 0.])

# The right way: hand cross_entropy the LOGITS, never probabilities.
logits = z.unsqueeze(0)          # shape (1, 3)
target = torch.tensor([0])       # true class index
print(F.cross_entropy(logits, target).item())  # 0.417030 — fused, stable
