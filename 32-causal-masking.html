Causal Masking: 4096 Signals Per Pass → the LLM hinge

🔮 Predict first You have one 4,096-token document. Training a next-token LLM on it, how many separate "predict the next token" learning signals do you get from a single forward pass — 1, about 15% of 4,096, or all 4,096? Commit before you scroll — you have watched LLMs generate token-by-token, and that intuition is about to mislead you.

The answer is all 4,096 — every one of the sequence's positions yields a valid "predict my successor" loss term from the same matmul, and that number is the whole reason LLM pretraining is affordable. The mechanism that makes it true, and not a cheat, is the causal mask below; the signals counter in the demo lands you on the number itself.

The blindfold, and what it buys

Train a model to predict the next token, and let it see the next token while it predicts, and it learns the trivial cheat: copy the token sitting one slot to the right. Zero loss, zero understanding. The causal mask is a blindfold that keeps training honest — position $i$ may attend only to itself and its predecessors, $j \le i$, never $j > i$.

The non-obvious part isn't the blindfold — it's what the blindfold buys. Because every position is now a legitimate, un-cheatable "predict my successor" problem, one sequence of length $S$ becomes $S$ training examples at once, all computed in a single parallel forward pass. That fact is section 3 below, and it is the largest single reason decoder-only pretraining scales the way it does.

The mask: additive, before softmax

The mask $M$ is $(S,S)$: zero where position $i$ is allowed to look at position $j$, $-\infty$ where it isn't. It is added to the scaled scores, before the softmax:

$$M_{ij} = \begin{cases} 0 & j \le i \\ -\infty & j > i \end{cases} \qquad\qquad A = \operatorname{softmax}_{\text{row}}\!\Big(\dfrac{QK^\top}{\sqrt{d_{\text{head}}}} + M\Big).$$

In practice $-\infty$ is torch.finfo(dtype).min — about $-3.39\times10^{38}$ in bfloat16 — so $\exp(\cdot)$ underflows the masked entries to exactly $0$ before the row is renormalized. Row $i$ ends up with $i{+}1$ live columns out of $S$: row $0$ sees only itself, row $S{-}1$ sees everything.

⚠️ Watch out — additive, not multiplicative (the real bug)

Additive-before-softmax renormalizes the survivors — every row still sums to exactly 1, because that renormalization is softmax's whole job. Multiplying the attention weights by a $0/1$ mask after the softmax skips that renormalization, and the row simply loses mass.

Take the course's frozen three-logit softmax from page 30, $z=[2.0,1.0,0.1] \Rightarrow \hat y = [\mathbf{0.659001},\,0.242433,\,0.098566]$. Imagine that's row 0 of a 3-token sequence, causally masked so row 0 may see only column 0:

  • Correct (additive, before softmax): only logit 0 survives the mask; softmax of a single value renormalizes to $[\mathbf{1.0},\,0,\,0]$ — row sum $= 1.0$.
  • Buggy (multiplicative, after softmax): zero out columns 1,2 of the already-computed $\hat y$: $[\mathbf{0.659001},\,0,\,0]$ — row sum $= 0.659001 \ne 1$. The kept weight is whatever the unmasked softmax happened to give column 0, not the honest "this is the only thing I can see."

Multiply the demo below into buggy mode and watch row 0's kept mass sit near $1/S$ instead of $1$ — the earlier the row, the worse the shrinkage, because early rows have the fewest live columns to begin with.

THE efficiency insight — $S$ signals per pass

Turn the mask's consequence into an equation. Because position $i$'s output depends only on $x_{\le i}$, a single forward pass over a length-$S$ sequence yields $S$ simultaneous next-token predictions, each a valid, un-cheatable loss term:

$$J = -\frac{1}{S}\sum_{i=1}^{S} \log p_\theta\big(x_{i+1} \mid x_{\le i}\big)$$

🔑 Key idea A 4,096-token training slice gives $S = \mathbf{4096}$ training signals from one pass [DER — direct consequence of the mask, not measured]. Contrast masked-language modeling (BERT), which masks $\approx 15\%$ of tokens and gets $\approx 0.15$ signals per token against causal LM's $1.0$: a $\approx\mathbf{6.7\times}$ data-efficiency gap per forward FLOP ($1/0.15 = 6.\overline{6}$) [$M$ for the $15\%$ figure — a stated hyperparameter, not this page's measurement]. This is the largest single piece of the honest "why decoder-only won" story (page 35).

The generation payoff — why causality is the KV cache

Turn the mask around and read it as a promise about the past, not just a restriction on the present. Position $i$ depends only on $j \le i$ — so once position $j$'s key and value are computed, no later token's arrival can ever change them. That is not an optimization; it's a fact the mask guarantees.

🔑 Key idea — the KV cache is a consequence, not a bolt-on Because earlier positions' $K,V$ are frozen by the mask, you compute each position's $K,V$ once and cache it. Every new generated token is a single query against the cached $K,V$ — not a full recompute of the sequence. Caching is only sound because causality guarantees the past can't change. Page 37 sizes exactly what this buys: 144 KiB/token for Qwen3-8B's GQA cache (constants.md §4).

The honest flip side: at inference the transformer goes sequential again, one new token at a time, and inherits RNN economics — bandwidth-bound decode, page 28's irony, quantified on the roofline (page 43). Training parallelism and generation's cache are the same fact, read in two directions.

Parallel training vs sequential inference — the distinction to nail

At training, all $S$ positions are computed in the same matmul. The mask constrains what each position may see; it says nothing about when that position is computed — they are all computed together, right now. That is the exact property that separates a transformer from an RNN (page 28): the RNN's sequential dependency lives in the computation itself; the transformer's causality lives only in visibility.

Only at decode — generating one new token at a time, each depending on the last — does it become sequential. That's not the mask's doing; it's the nature of autoregressive generation. The mask is what makes the sequential part cheap once you're in it (the cache above); it is not what causes the sequentiality.

Five traps

⚠️ Watch out

(a) "Causal masking makes the model slower — it's a limitation." The opposite: it is the enabling trick for parallel training. Without it you'd need RNN economics at training time too. Fix with the 4,096-signals-per-pass fact above.

(b) "The mask is applied to the output — to $A$, after softmax." It's applied to the logits, before softmax, additively. Apply it after and rows stop summing to 1 (the worked box above).

(c) "Causal = the model reads left-to-right, one token at a time." Only at inference. At training, all positions are computed in one matmul; the mask limits visibility, not timing.

(d) "The KV cache is a separate optimization you bolt on later." It is a direct consequence of causality — past $K,V$ are frozen by the mask, which is the only reason caching is correct at all.

(e) "Diffusion U-Nets are causal too." No — image self-attention is bidirectional and unmasked (page 31's fork). Causality is a property of time-ordered generation, not of attention in general.

The code that proves it

Try it — code/causal_mask.py (1) Calls F.scaled_dot_product_attention(..., is_causal=True) and, from a manual reference implementation, asserts the weight matrix is strictly-lower-triangular with every row summing to 1. (2) Reproduces the multiply-after-softmax bug and prints early rows summing to $<1$. (3) From one forward pass over a length-$S$ sequence, computes the full per-position cross-entropy vector — $S$ loss terms, one pass — and prints $S$. (4) A tiny caching demo: generates 3 tokens and asserts the cached $K,V$ for earlier positions are bit-identical across steps.
Python
import math
import torch
import torch.nn as nn
import torch.nn.functional as F

torch.manual_seed(32)
B, S, d_head, V = 1, 10, 16, 50
Q = torch.randn(B, S, d_head)
K = torch.randn(B, S, d_head)
Vv = torch.randn(B, S, d_head)

# (1) SDPA's fused kernel gives no weight matrix back, so build the reference
# manually to check it against, exactly what SDPA computes under is_causal=True.
def manual_causal_attention(Q, K, Vv):
    scores = Q @ K.transpose(-1, -2) / math.sqrt(Q.shape[-1])   # (B,S,S)
    mask = torch.triu(torch.ones(S, S, dtype=torch.bool), diagonal=1)          # j > i
    scores = scores.masked_fill(mask, torch.finfo(scores.dtype).min)  # additive, before softmax
    A = F.softmax(scores, dim=-1)
    return A @ Vv, A

O_manual, A = manual_causal_attention(Q, K, Vv)
O_sdpa = F.scaled_dot_product_attention(Q, K, Vv, is_causal=True)
assert torch.allclose(O_manual, O_sdpa, atol=1e-5), "SDPA and the manual reference must agree"
assert torch.allclose(A.sum(dim=-1), torch.ones(B, S), atol=1e-5), "every row must sum to 1"
for i in range(S):
    assert torch.all(A[0, i, i + 1:] == 0), f"row {i} leaked into the future"
print("(1) causal weights: strictly-lower-triangular, every row sums to 1 — OK")

# (2) the multiply-after-softmax bug: unmasked softmax, THEN zero the future.
scores_full = Q @ K.transpose(-1, -2) / math.sqrt(Q.shape[-1])
A_full = F.softmax(scores_full, dim=-1)                     # rows sum to 1 here
keep = ~torch.triu(torch.ones(S, S, dtype=torch.bool), diagonal=1)
A_buggy = A_full * keep                                          # multiplicative, no renormalize
row_sums = A_buggy.sum(dim=-1)[0]
assert row_sums[0] < 1.0, "row 0 must lose mass under the multiplicative bug"
print("(2) buggy row sums (first 3):", row_sums[:3].tolist())

# (3) one forward pass, S loss terms.
logits = torch.randn(B, S, V)                                  # stand-in LM head output
targets = torch.randint(0, V, (B, S))                            # x_{i+1}, shifted
per_position_ce = F.cross_entropy(logits.view(-1, V), targets.view(-1), reduction="none")
assert per_position_ce.numel() == S
print(f"(3) one forward pass over S={{S}} tokens yields {{per_position_ce.numel()}} loss terms")

# (4) caching: generate 3 tokens; earlier positions' K,V must not change.
cache_K, cache_V = [], []
seq = torch.randn(B, 4, d_head)     # a 4-token prompt
for step in range(3):
    k_new, v_new = seq.clone(), seq.clone()          # identity "projection" — the caching logic is what's under test
    cache_K.append(k_new.clone()); cache_V.append(v_new.clone())
    next_tok = torch.randn(B, 1, d_head)
    seq = torch.cat([seq, next_tok], dim=1)              # sequence grows by one
for step in range(1, 3):
    assert torch.equal(cache_K[0], cache_K[step][:, :4]), "earlier K must be bit-identical across steps"
print("(4) cached K for the first 4 positions is bit-identical across 3 generation steps — OK")
