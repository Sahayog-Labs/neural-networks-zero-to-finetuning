Positional Encoding and RoPE

Attention, you just learned, is a set operation wearing a sequence costume — it cannot tell "dog bites man" from "man bites dog." This page is how we tell it the order back. The good version doesn't add position; it rotates for it, and the score falls out depending only on the gap between two tokens. It is the most quietly beautiful trick in the architecture, and it is one 2×2 matrix you already know.

🧵 Thread [THREAD:dot-product — the atom, beat 5] · [THREAD:chain-rule — the verb] · [THREAD:Qwen3-8B — the anchor]. The RoPE score is, once more, a dot product — the same needle from page 27, now measured between two rotated vectors, so it reads out the angle between their positions. And the payoff — relative position appears without the model ever computing a difference — is the chain rule's cousin: the algebra does the subtraction, you never write it. We anchor everything on Qwen3-8B's real config line, rope_theta: 1000000, which you have scrolled past and never decoded. By the end it is obvious.
🔮 Predict first Take a trained transformer and delete its positional encoding entirely. Now shuffle the words of a sentence and feed them in. Does the output (a) change unpredictably, (b) come out shuffled the same way, or (c) stay identical per token? Commit to one before you scroll — the three-line proof below settles it.

The problem: attention has no idea what order anything is in

You met the score matrix as $S = QK^\top$ — a table of dot products, computed all at once. Look at what is missing from it: nowhere does the position of a token enter. $\mathbf q_i$ is a projection of $\mathbf x_i$; $\mathbf k_j$ is a projection of $\mathbf x_j$; the score $\mathbf q_i\cdot\mathbf k_j$ knows the two contents and nothing about where they sat. The FFN is position-wise too — it hits every position with the same weights independently. So the whole transformer, minus positional encoding, is a set function in a sequence costume.

Don't take that on faith — it's three lines. Let $P$ be an $S\times S$ permutation matrix (one $1$ per row and column, zeros elsewhere; $P^\top P = I$). Permuting the input rows permutes each projection: $Q' = PQ$, $K' = PK$, $V' = PV$. Then

$$S' = Q'K'^\top = (PQ)(PK)^\top = P\,(QK^\top)\,P^\top = PSP^\top,$$

row-wise softmax commutes with a consistent row/column permutation, so $A' = PAP^\top$, and finally

$$A'V' = (PAP^\top)(PV) = PA\,(P^\top P)\,V = PAV. \qquad\blacksquare$$

🔑 The PREDICT answer: permutation-equivariant $A'V' = P\,(AV)$ — the outputs come out shuffled exactly the same way the inputs were (answer (b)). Per token, nothing changed: token "bites" gets the identical output vector whether it was 2nd or 5th. So a transformer with no positional signal literally cannot distinguish "dog bites man" from "man bites dog." A CNN gets order for free from kernel geometry; an RNN gets it free from loop order. The transformer threw both away — that was the weak-inductive-bias bet of page 25 — and now it has to buy position back. The rest of this page is the three ways people have paid.
🗺️ Translation note (read once, applies all page) We index sequence position by $i,j$ (notation §3); the RoPE papers write $m,n$. We reserve $\theta$ for learnable parameters and write a geometric angle as $\vartheta$; the papers overload $\theta$ for both the angle and the frequency base. So the base you will see in configs as rope_theta we call $b$, and the per-pair rotation frequency we write $\varphi_p$ (varphi, distinct from the activation $\phi$). Same math, translated symbols — that is the durable lesson, not a symbol you're failing to know.

Three ways to pay: absolute, learned, rotated

(1) Sinusoidal absolute (2017). Add a fixed vector to each token that encodes its position as a stack of sines and cosines at geometrically-spaced frequencies:

$$PE_{(pos,\,2p)} = \sin\!\Big(\frac{pos}{b^{\,2p/d}}\Big),\qquad PE_{(pos,\,2p+1)} = \cos\!\Big(\frac{pos}{b^{\,2p/d}}\Big).$$

It's a continuous binary counter: the fast dimensions are the ones' digit, the slow dimensions the millions' digit, and together they name a position uniquely. It works, but it has two aches. It is absolute — the model usually wants "you're three tokens behind me," which it must reconstruct from two absolute encodings — and it is added, so position competes with meaning for room in the same vector. Panel A below is exactly this zebra; drive its base and watch the slow columns barely move.

(2) Learned absolute (BERT, GPT-2). Skip the math; just learn a lookup table of shape $(S_{\max}, d)$, one row per position. Simple, and it works to $S_{\max}$. But it cannot extrapolate at all: position 2049 in a model trained to 2048 is a literally untrained, random vector. This is precisely why GPT-2 could not be context-extended after the fact and RoPE-based models can be.

(3) RoPE — 2026's default, and it's the rotation matrix you already know

🔑 The whole idea in one sentence RoPE doesn't add position — it rotates the vector by an angle proportional to its position. Two tokens' dot product then depends only on the angle between them, which is the difference of their positions. You get relative position for free, out of the geometry, without ever computing a difference.

Mechanism. Split $\mathbf q\in\mathbb R^{d_{\text{head}}}$ into $d_{\text{head}}/2$ consecutive 2-D pairs. Rotate pair $p$ by an angle equal to the position $i$ times a per-pair frequency $\varphi_p$:

$$\begin{pmatrix} q'_{2p} \\ q'_{2p+1}\end{pmatrix} = \underbrace{\begin{pmatrix}\cos i\varphi_p & -\sin i\varphi_p \\[2pt] \sin i\varphi_p & \ \ \cos i\varphi_p\end{pmatrix}}_{\displaystyle \text{the }2\times2\text{ rotation you know}} \begin{pmatrix} q_{2p} \\ q_{2p+1}\end{pmatrix}, \qquad \varphi_p = b^{\,-2p/d_{\text{head}}}.$$

Applied to $\mathbf q$ and $\mathbf k$ only — never $\mathbf v$, never the residual stream. That restraint is the elegant part: position touches the two vectors whose dot product needs to be position-aware, and leaves everything else alone. For Qwen3-8B, $d_{\text{head}}=128$ gives 64 rotation pairs, and the whole scheme has zero learnable parameters — it's pure geometry, computed on the fly at every layer.

Why it works — write the identity out, because the identity is the justification

A rotation by $i\varphi$ followed by the transpose of a rotation by $j\varphi$ is a rotation by $(j-i)\varphi$. In matrix form, with $R_i$ the block-diagonal rotation for position $i$, rotations compose so that $R_i^\top R_j = R_{j-i}$. Slide that into the score:

$$\big\langle R_i\mathbf q,\; R_j\mathbf k\big\rangle = \mathbf q^\top R_i^\top R_j\,\mathbf k = \mathbf q^\top R_{\,j-i}\,\mathbf k.$$

Read the right-hand side: the positional part of the score is $R_{j-i}$ — it depends on the gap $j-i$ and on nothing else positional. You injected absolute position ($i$ into the query, $j$ into the key, independently and cheaply) and got relative position ($j-i$) out, for free, because the two rotations compose into one. You never computed $j-i$ — the algebra performed the subtraction. And because a rotation preserves length ($\lVert R_i\mathbf q\rVert = \lVert\mathbf q\rVert$), position never steals magnitude from meaning — the ache that sinusoidal-added encodings have. This is genuinely beautiful; Panel B lets you watch the two readouts agree to eight decimals while you drag the positions around.

What the base $b$ actually buys — carry the arithmetic

The frequencies span from fast to slow. The fastest pair, $p=0$, has $\varphi_0 = b^{0} = 1$ radian per token — wavelength $2\pi \approx 6.28$ tokens, regardless of the base. The slowest pair, $p=63$, is where the base bites: $\varphi_{63} = b^{-126/128}$. Plug in the two bases you'll meet:

✍️ The slowest pair, two bases [DER] Classic $b = 10{,}000$: $\varphi_{63} = 10000^{-126/128} = 10^{-3.9375} \approx 1.155\times10^{-4}$ rad/token $\;\Rightarrow\;$ wavelength $\dfrac{2\pi}{\varphi_{63}} \approx 5.44\times10^{4}$ tokens.
Qwen3-8B's $b = 1{,}000{,}000$ [VP]: $\varphi_{63} = 10^{6\times(-0.984375)} = 10^{-5.90625} \approx 1.24\times10^{-6}$ rad/token $\;\Rightarrow\;$ wavelength $\approx 5.06\times10^{6}$ tokens.
Same fast pair (6.28 tokens); the slow end stretched by exactly the base ratio. Panel C plots both and lets you slide a "your context length" line across them.
📐 The rule the base enforces Raising the base stretches the low-frequency end and leaves the high-frequency end alone. The reason it matters: the slowest pair must not complete a full rotation inside your context window, or two positions a wavelength apart become indistinguishable in that pair — they alias. At $b=10^4$ the slow pair wraps around every ~54k tokens (fine at 2k, marginal at 128k); at $b=10^6$ its wavelength is ~5.06M, and Qwen3-8B's max_position_embeddings = 40960 [VP] sits comfortably inside it. That is the entire meaning of "increase rope_theta for long context" — the config line you've seen and never decoded.

⚠️ The rotate_half gotcha that costs people days The paper pairs adjacent dimensions $(0,1),(2,3),\dots$. HuggingFace's rotate_half instead splits the vector in half and pairs across: $(0,\,d_{\text{head}}/2),\,(1,\,d_{\text{head}}/2{+}1),\dots$ The two conventions are equivalent up to a dimension permutation the learned weights absorb — as long as you pick one and stay in it. Mix them with a pretrained checkpoint and you get output that looks exactly like a training bug (garbage that's almost coherent), with nothing in the stack trace to point at. This is a genuine days-long trap; code/rope_identity.py shows one convention matching the checkpoint and the other producing garbage from the same weights.
🔬 Deep dive: long-context extension is all the same move — rescale the frequency spectrum

Qwen3-8B ships rope_scaling: null [VP] — it is natively 40,960 and needs no trick. But when you do push a model past its trained length, every method is the same idea in different clothing: make the angles seen at position 100k look like angles the model saw in training.

  • Position Interpolation (PI): squash all positions into the trained range — divides every angle by the extension factor. Cheap, but it hurts local resolution (adjacent tokens get closer angles).
  • NTK-aware: raise the base instead — stretches the low frequencies, leaves the high ones roughly put, so local detail survives. This is the "just bump rope_theta" folk move, made principled.
  • YaRN: do it per band — interpolate the low-frequency pairs, leave the high-frequency pairs, and add a small attention-temperature correction. Reaches ~128k with far less fine-tuning than PI.

None of this changes the mechanism you just learned; it only rescales $\varphi_p$. The unifying sentence: long-context RoPE is spectrum engineering.

Four traps

⚠️ "Positional encoding tells the model the order." It makes order recoverable, not mandatory. Nothing forces a head to use it — some heads are demonstrably position-agnostic and attend purely by content. The encoding supplies the signal; the model decides, per head, whether to listen.
⚠️ "RoPE is added to the input embeddings." That is sinusoidal/learned — added once, at the bottom. RoPE is applied to $Q$ and $K$ only, inside every attention layer, every forward pass, and never to $V$ or the residual stream. In code it sits between the projection and the score matmul: q, k = apply_rope(q, k, positions), then scores = q @ k.transpose(-2,-1). If you're adding it to embeddings, you've built the wrong thing.
⚠️ "RoPE gives unlimited context." It extrapolates better than a learned table, but it degrades before it fails — attention gets mushy past the trained length well before it breaks outright. That gap between "degrades" and "fails" is the entire reason the PI / NTK / YaRN industry exists. RoPE buys graceful decline, not a free lunch.
⚠️ "A decoder-only model strictly needs positional encoding." (genuinely interesting) The causal mask alone breaks permutation-equivariance — token $i$ sees exactly $i$ predecessors, so it can count. Reportedly, "NoPE" decoder-only models do learn a workable notion of position from the mask and can function. Everyone ships RoPE anyway, because NoPE is reported to be worse at long context — but "the thing everyone calls necessary isn't strictly necessary" is worth holding: it tells you the mask itself carries order information, which is a real fact about causal attention (page 32).
Try it — code/rope_identity.py Build $R_i$ for a random $\mathbf q,\mathbf k$ and assert allclose(⟨R_i q, R_j k⟩, qᵀ R_(j−i) k) across random $i,j$ — RoPE depends only on the relative position. Then apply HF's rotate_half versus the adjacent-pair convention to the same pretrained q_proj slice and watch one match the checkpoint and the other emit garbage. Uses rope_theta = 1e6 (constants §1.1).
Python
import torch, math

def rope_angles(d_head, base=1_000_000):        # rope_theta = 1e6 (constants §1.1)
    p = torch.arange(0, d_head, 2) / d_head
    return base ** (-p)                          # phi_p, shape (d_head/2,) = (64,)

def R_adjacent(v, pos, phi):                     # paper: pair (0,1),(2,3),...
    ang = pos * phi; c, s = torch.cos(ang), torch.sin(ang)
    x, y = v[..., 0::2], v[..., 1::2]
    out = torch.empty_like(v)
    out[..., 0::2], out[..., 1::2] = c*x - s*y, s*x + c*y
    return out

def rotate_half(x):                             # HF: split-half, pair (0,64),(1,65),...
    h = x.shape[-1] // 2;  return torch.cat((-x[..., h:], x[..., :h]), dim=-1)
def R_hf(v, pos, phi):
    ang = torch.cat((pos*phi, pos*phi), dim=-1)
    return v * torch.cos(ang) + rotate_half(v) * torch.sin(ang)

d_head = 128; phi = rope_angles(d_head)

# PART 1 — the identity holds for EITHER convention, on its own
q, k = torch.randn(d_head), torch.randn(d_head); i, j = 137, 42
for R in (R_adjacent, R_hf):
    lhs = (R(q, i, phi) * R(k, j, phi)).sum()   # ⟨R_i q, R_j k⟩
    rhs = (q * R(k, j - i, phi)).sum()          # qᵀ R_(j−i) k
    assert torch.allclose(lhs, rhs, atol=1e-4) # the score knew only the gap

# PART 2 — same weights, two conventions: one matches the checkpoint, one is garbage
X  = torch.randn(24, 4096)                       # a token sequence (hidden=4096)
Wq = torch.randn(d_head, 4096) / 64; Wk = torch.randn(d_head, 4096) / 64
pos = torch.arange(1, 25) * 173
def scores(R):                                  # RoPE'd attention score matrix
    Q, K = X @ Wq.T, X @ Wk.T
    Qr = torch.stack([R(Q[t], int(pos[t]), phi) for t in range(24)])
    Kr = torch.stack([R(K[t], int(pos[t]), phi) for t in range(24)])
    return (Qr @ Kr.T) / math.sqrt(d_head)
ref = scores(R_hf)                             # the checkpoint was trained under HF's convention
assert torch.allclose(ref, scores(R_hf))       # matching convention → reproduces it exactly
flip = (ref.softmax(-1).argmax(-1) != scores(R_adjacent).softmax(-1).argmax(-1))
print(f"wrong convention flips {flip.float().mean()*100:.0f}% of attended tokens")  # ~92%
