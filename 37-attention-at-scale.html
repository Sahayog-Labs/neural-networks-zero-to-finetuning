Part III — Architecture

Attention at Scale: $O(S^2)$, FlashAttention & the KV Cache

Three different costs hide inside one softmax(QKᵀ/√d)V, and they have three different fixes. Conflate them and you will optimize the wrong one. This is the last page of the trunk before Adaptation — and the final beat of the memory-budget thread.

You have the transformer block assembled (page 35) and counted to the byte on Qwen3-8B (page 36). One question is left before we leave architecture behind: what does attention cost when the sequence gets long? The famous answer — "it's $O(S^2)$" — is true, incomplete, and usually points at the wrong bottleneck. There are really three costs stacked inside the same formula, and untangling them is the whole page.

🔮 Predict, before you scroll Qwen3-8B advertises a 40,960-token context. Its single-layer attention score matrix at that length, in fp32, is about — 200 MB, 2 GB, or 200 GB? Commit to one. (There is a real GPU on your desk with 128 GB of unified memory; does your answer fit in it?)

Translation note (notation §3.2): this course writes $S$ for sequence length and $i,j$ for positions. Papers and configs write $T$, $L$, $N$, $n$, seq_len, or max_length for $S$, and $t$, $i$, $n$, or $j$ for position — four rival letters for the one thing. $H$ is the number of query heads; $H_{kv}$ is the number of key/value heads (num_key_value_heads), and the distinction is about to earn its keep.

✍️ The 214.7 GB — resolving the predict The score matrix $S = QK^\top$ is $(S, S)$ per head, and there are $H$ heads: $$H\,S^2\,b = 32 \times 40{,}960^2 \times 4\ \text{B} = \mathbf{214.7\ GB}\ \text{per layer, fp32.}$$ So the answer was 200 GB — and it is per layer, of which there are 36. A single layer's score matrix at the model's own advertised context does not fit in any GPU ever made. And yet the model runs. Hold that paradox; it is what FlashAttention exists to dissolve, two sections down.

The attention and FFN FLOP counts come from the matmul rule $2mnp$ for an $(m,n)\!\times\!(n,p)$ product (multiply and add counted separately, as vendors do — constants §5). Attention is two matmuls per head, $QK^\top$ and $AV$, each $2S^2 d_{\text{head}}$ summed over heads $\Rightarrow 4S^2 d$. The FFN is three matrices (gate, up, down), each $2S\,d\,d_{\text{ff}} \Rightarrow 6S\,d\,d_{\text{ff}}$.

The crossover: when is $O(S^2)$ actually your problem?

Attention FLOPs grow with $S^2$; FFN FLOPs grow with $S$. So there is a crossover length below which the FFN — not attention — dominates compute. Set them equal:

$$4S^2 d > 6S\,d\,d_{\text{ff}} \quad\Longleftrightarrow\quad S > \tfrac{3}{2}\,d_{\text{ff}} = 1.5 \times 12{,}288 = \mathbf{18{,}432}\ \text{tokens.}$$

🔑 Key idea Below 18,432 tokens, the FFN dominates attention in FLOPs. At ordinary chat lengths, "it's $O(S^2)$" is the wrong bottleneck — attention isn't where your compute goes. The $d$ cancels: the crossover depends only on the FFN expansion ratio, which is why it's $\tfrac{3}{2}d_{\text{ff}}$ and nothing else.

Mitigation 1 — FlashAttention: fix the memory, not the FLOPs

The 214.7 GB never actually gets allocated. Naive attention materializes the full $(S,S)$ score matrix in HBM before the softmax touches it. FlashAttention never does. It tiles $Q$, $K$, $V$ into blocks that fit in on-chip SRAM, and carries a running maximum and a running normalizer so the softmax corrects itself as blocks stream in — the online-softmax (streaming log-sum-exp) identity, which is algebra, not approximation.

shape ribbon — what FlashAttention refuses to build
X            (B, S, d=4096)                      # residual stream in
W_q          (H*d_head=4096, d=4096)             # (out, in) — ALWAYS
W_k, W_v     (H_kv*d_head=1024, d=4096)          # (out, in) — note H_kv, not H
Q = X @ W_q.T   -> reshape/transpose             (B, H=32,    S, d_head=128)
K = X @ W_k.T   -> reshape/transpose             (B, H_kv=8,  S, d_head=128)   # 8 KV heads shared by 32 Q heads
V = X @ W_v.T   -> reshape/transpose             (B, H_kv=8,  S, d_head=128)
scores = Q @ K.T / sqrt(d_head)                  (B, H=32, S, S)   <-  the (S,S) monster: 214.7 GB/layer @ S=40960
A = softmax(scores + M, axis=-1)                 (B, H, S, S)      # rows sum to 1, columns do not
out = A @ V                                       (B, H, S, d_head)
                                                 FlashAttention computes `out` WITHOUT ever storing `scores`.

Memory drops from $O(S^2)$ to $O(S)$; the FLOP count stays the same, or rises slightly (the backward pass recomputes tiles instead of storing them). It is faster anyway, because attention is memory-bandwidth-bound, not compute-bound — the GPU spends its time moving that giant matrix, not multiplying it.

🔑 The sentence to keep FlashAttention made attention faster by making it do more arithmetic. That is only paradoxical if you think the GPU's problem is arithmetic. Its problem is memory.
🔬 Deep dive: FlashAttention-4 and Blackwell's asymmetry (2026)

FlashAttention-4 (paper released 5 March 2026; flash-attn-4 on PyPI) targets Blackwell — including the GB10 in your DGX Spark. Blackwell scaled its tensor cores hard but did not scale the special-function unit (SFU) that computes exp() at the same rate. So the exponential inside the softmax became a relative bottleneck. FA4's answer is a co-design move you can appreciate directly: it computes exp() with a polynomial evaluated on the ordinary FMA (fused multiply-add) units instead of waiting on the SFU, keeping the tensor cores fed. The lesson generalizes past this one kernel — the fast path is whichever unit isn't the bottleneck, and which unit that is changes with every hardware generation. This is the same "measure your own box" discipline the roofline page (43) turns into an exercise. (Vendor throughput multipliers here are reported figures, not something this course freezes as fact.)

Mitigation 2 — GQA: fix the KV cache, not the score matrix

FlashAttention fixes the training/prefill activation memory. It does nothing for the other memory cost — the KV cache at decode time. These are orthogonal, and both ship in every 2026 model. The cache exists at all only because causal masking (page 32) freezes each position's past $K,V$: once computed, they never change, so you keep them instead of recomputing. Here we count the bytes. ✍️ Worked — the cache, byte by byte [DER, constants §4] Per token, per layer: $2 \times 8 \times 128 \times 2 = \mathbf{4{,}096\ B = 4\ KiB}$.
Per token, all 36 layers: $4{,}096 \times 36 = 147{,}456\ \text{B} = \mathbf{144\ KiB/token}$.
At full context $S = 40{,}960$: $147{,}456 \times 40{,}960 = 6{,}039{,}797{,}760\ \text{B} = \mathbf{5.625\ GiB}$.
The counterfactual that shows why GQA is universal: if this were plain multi-head attention ($H_{kv} = H = 32$), the per-token cost would be $576\ \text{KiB}$ and the full-context cache would be $\mathbf{22.50\ GiB}$. GQA's ratio $H/H_{kv} = 32/8 = 4$ saves exactly 4× and recovers 16.88 GiB.
📐 Why 4× matters, in GiB Qwen3-8B's bf16 weights are 15.26 GiB (16.38 GB — constants §1.1, §4). The MHA counterfactual cache, 22.50 GiB, would exceed the entire model at full context. With GQA it is 5.625 GiB — a third of the weights. That gap is why 8 KV heads shared by 32 query heads is universal in 2026: without it, the thing you keep around to avoid recompute would outweigh the thing you are running. (Comparison done in GiB throughout — never mix GiB against GB when sizing to the box; constants §0.)

Mechanically: $H = 32$ query heads share $H_{kv} = 8$ key/value heads — four query heads per KV head. MHA is $H_{kv} = H$; MQA (multi-query) is $H_{kv} = 1$; GQA interpolates, and GQA-8 is the sweet spot Qwen3 ships. 🔬 Deep dive: mitigations 3–5 — MLA, sliding-window/hybrid, FlexAttention

MLA (multi-head latent attention). The DeepSeek lineage (and, as of 2026, Kimi K2 / GLM-5 / Ling 2.5) caches a single low-rank latent per token and decompresses $K,V$ on the fly. It trades compute for memory — the opposite direction from GQA. Learn the idea, not the projection equations: shrink what you store, pay a little arithmetic to reconstitute it.

Sliding-window / hybrid. Most layers attend only to a local window (e.g. SWA-4096), a few attend globally, commonly in a 3:1 local:global ratio; recurrent-style linear-attention hybrids (Qwen3-Coder-Next's Gated DeltaNet, 262k context) push this further. The cache stops growing with $S$ on the windowed layers.

FlexAttention (PyTorch 2.5+). Write your mask/score modification as a Python score_mod/mask_mod and let torch.compile fuse it into a FlashAttention-style kernel. Honest caveats: the kernel options are not yet a public API, there are documented numerical discrepancies vs SDPA, and it can silently fall back to a dense path — profile to confirm. Do not build a core code path on it. Default to F.scaled_dot_product_attention (stable, dispatches to a FlashAttention backend); show the naive implementation first for understanding; keep FlexAttention as the escape hatch.

⚠️ Four things people get wrong about "attention is expensive" (a) "FlashAttention is an approximation / a kind of linear attention." No — it is numerically exact (up to floating-point reassociation). It changes where data lives, not what is computed; the online-softmax identity is algebraic, not a truncation.
(b) "FlashAttention makes attention $O(S)$." It makes the memory $O(S)$. The FLOPs stay $O(S^2)$ — which is exactly why 128k context is possible but still expensive.
(c) "GQA is a quality-hurting approximation." It is an architectural choice made before pretraining: the model is trained with 8 KV heads and never had 32, so nothing is being approximated away. (Uptraining an existing MHA model down to GQA after the fact is a different operation and does cost a little quality — but that is not what Qwen3 did.)
(d) "$O(S^2)$ is why long context is hard." Below 18,432 tokens the FFN dominates FLOPs, and at decode the binding constraint is KV-cache memory and bandwidth, not attention FLOPs. "It's $S^2$" is usually the wrong bottleneck.

# The KV-cache ledger, self-contained — the arithmetic the demo draws.
cfg = dict(L=36, H=32, H_kv=8, d_head=128, b=2, S=40960)  # Qwen3-8B, bf16 KV

def kv_bytes(L, H_kv, d_head, b, S):
    return 2 * S * L * H_kv * d_head * b        # 2 = K and V

per_tok_layer = kv_bytes(1, cfg["H_kv"], cfg["d_head"], cfg["b"], 1)
per_tok_all   = kv_bytes(cfg["L"], cfg["H_kv"], cfg["d_head"], cfg["b"], 1)
full          = kv_bytes(cfg["L"], cfg["H_kv"], cfg["d_head"], cfg["b"], cfg["S"])
mha           = kv_bytes(cfg["L"], cfg["H"],    cfg["d_head"], cfg["b"], cfg["S"])

print(per_tok_layer, "B/token/layer   ->", per_tok_layer // 1024, "KiB")   # 4096 -> 4 KiB
print(per_tok_all // 1024, "KiB/token (all layers)")                    # 144 KiB
print(round(full / 2**30, 3), "GiB (GQA)  vs ", round(mha / 2**30, 2), "GiB (MHA)")  # 5.625 vs 22.5
