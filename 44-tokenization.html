The one-sentence frame

Hold onto this sentence; the rest of the page just unpacks it:

The model sees integers drawn from a fixed dictionary of about 150,000 word-fragments, itself learned by a greedy compression algorithm run once over uncurated text.

For Qwen3-8B that dictionary has exactly $V = 151{,}936$ entries [VP, constants §1.1] — one of the fourteen reflex numbers you'll meet again and again in this track. Nothing about the number is arbitrary in the sense of "someone typed it in"; it's a design choice with a real cost, which the last section of this page prices out.

The BPE mechanism

A tokenizer is a function $\tau$ that turns a string of raw bytes into a sequence of integer IDs:

$$\tau : \Sigma^{*} \to \{0, \dots, V-1\}^{*}, \qquad |\Sigma| = 256$$

SymbolIsShape / rangeFrom
$\tau$the trained tokenizer's encode functionstring → sequence of IDsthe object this page defines
$\Sigma$the byte alphabet$|\Sigma| = 256$ — every possible byte valuefixed, not learned
$V$vocabulary size151,936 for Qwen3-8Bconstants §1.1 [VP]

$\tau$ itself is learned by Byte-Pair Encoding, a four-step loop that never looks at grammar or meaning — only frequency:

  1. Count. Over the whole training corpus, count how often every adjacent pair of symbols occurs.
  2. Merge. Take the single most frequent pair and fuse it into one new symbol.
  3. Record. Append that pair, in order, to a growing ordered merge list.
  4. Repeat until the vocabulary reaches $V$ entries.
📐 The under-taught point Encoding a new sentence is not a search for the best possible split of that sentence. It is a mechanical replay of the merge list, in the exact order it was learned, on data the tokenizer has never seen. Whichever merge sits earliest in the list always wins, whether or not a "better" split exists downstream. Two tokenizers trained on different corpora will chop the same sentence differently, and neither is more correct — they just learned different schedules.

Compression ratio, and the worked number

Call $\rho$ the characters-per-token compression ratio. For English prose, $\rho \approx 4$ [EST, constants §9.5] — an estimate, not a measured constant, and the page says so every time it uses it. For code and non-Latin scripts, $\rho$ drops to roughly 1.5–2.5 [EST, constants §9.5]: the same amount of information costs more tokens.

✍️ Worked example — [EST], built on ρ≈4 An average English word runs about 5.2 characters including its trailing space. A 500-word document is then about $500 \times 5.2 = 2{,}600$ characters, and at $\rho \approx 4$ chars/token that's $$2{,}600 / 4 = \mathbf{650}\ \text{tokens}.$$ That's roughly 1.3 tokens per word — the rate the quiz below asks you to reuse. Because $\rho$ itself is an estimate, so is 650; the label travels with the number.

Byte-level BPE (the Qwen/GPT lineage) differs from the older SentencePiece-style word-vocabularies in a way that matters more than it sounds: because every one of the 256 possible bytes is already a valid starting symbol, a byte-level tokenizer can never emit an <UNK> token. Any input, in any language, in any encoding, always has a valid byte-level encoding. The older word-vocabulary approach had to reserve a special "unknown" symbol for anything outside its fixed word list, and silently threw information away whenever a rare word appeared.

A second, quieter fact: whitespace is not stripped before tokenization. A leading space lives inside the token that follows it, so " the" and "the" are different byte sequences and — almost always — different tokens with different IDs. This single fact causes a large fraction of the chat-template bugs you'll meet on page 48; the demo below makes it visible by rendering every space as a middle-dot.

Why $V$ matters downstream

The vocabulary size isn't free. Every one of Qwen3-8B's $V$ token IDs needs a row in the input embedding table and, since Qwen3 does not tie its embeddings, a separate row in the output lm_head. Both are $V \times d$ matrices, $d = 4096$ being the residual-stream width you already own from the trunk:

shapes
embed_tokens          (V=151936, d=4096)     # lookup table, input side
lm_head.weight        (V=151936, d=4096)     # untied — Qwen3 does NOT tie weights
151936 * 4096 = 622,329,856                  # exact, each — constants §1.2 [DER]
2 * 622,329,856 = 1,244,659,712 = 15.20% of P   # constants §1.3 [DER]

[THREAD:Qwen3-8B beat: tokenizer] — this is the first appearance of $V$ in the LLM track. It recurs as the exact output dimension of the cross-entropy loss on page 45, and as the 622 MB logits tensor you'll meet again on pages 46 and 49. Trunk callback: the softmax-over-classes math itself is not new — it's the same cross-entropy you derived on the trunk's logs/softmax/cross-entropy page, just with $V = 151{,}936$ classes instead of a handful.

⚠️ "Tokens are words" They're word-fragments. Sometimes a token is a whole common word (" the"); often it's less ("straw" + "berry"); occasionally, for a frequent phrase, it's more than one English word. Whitespace and case matter: "The", " The", and "the" can all be three different tokens.
⚠️ "I can just add tokens to the vocab" You can — resize_token_embeddings() is one line. But the new rows are randomly initialized and completely untrained. Naively extending the vocabulary degrades the model until you actually train those new embeddings; it is not a free lunch, and it is not something a LoRA on the existing weights will fix for you.

Translation note: this course writes $V$ for vocabulary size. Papers and config files write vocab_size, and occasionally $K$; they mean the same integer (notation §3.2).
